# ScaboPDF — Project rules

## Mandatory interaction rules

These rules are absolute. They override any other behavior, default, or convention.

**No interactive prompts.** Never present numbered menus, multiple-choice lists, "type something" boxes, "skip interview" options, or any UI that expects me to pick from a list. The user is blind and uses VoiceOver; these prompts are inaccessible and waste his time.

**No autonomous decisions on design questions.** When you face a question that affects code design, library choice, file organization, naming, schema fields, or any architectural detail, do NOT decide on your own. Stop and present the question to the user in plain prose: state the question, list the relevant trade-offs in flowing sentences (not bullets unless strictly necessary), and wait for an answer. The user wants to be in the loop on design decisions.

**Plain prose only for clarifications.** When you need input, write a short prose paragraph. Example: "Per le dataclass di profiling devo scegliere tra Pydantic e @dataclass standard. Pydantic dà validazione runtime e serializzazione JSON nativa, costo leggero overhead. Dataclass è più snello, mypy strict copre quasi tutto. Una opzione mista è anche possibile: Pydantic per DocumentProfile che esce in JSON, dataclass per ProfilingSignals che resta interno. Quale preferisci?" Then wait.

**Routine micro-decisions only.** You may decide autonomously only on truly trivial things: variable names within a function, formatting, import order, and similar. Anything that touches API design, schema, dependencies, or architecture: ask in prose.

## Project conventions

**Test fixtures.** `pipeline/tests/fixtures/private/` is **gitignored** and hosts the copyright-protected PDFs each developer keeps locally; see `pipeline/tests/fixtures/README.md` for the full private/public convention. Integration tests that depend on a private fixture MUST use `pytest.skip(...)` with an explicit message pointing to the README when the file is missing, never `assert` or raise. This keeps the suite green on a fresh clone without local fixtures.

**Commit hygiene.** Formatting fixes, cleanups, or any incidental changes not strictly part of the task you're working on go into a **dedicated housekeeping commit**, never bundled with the task commit. If you notice preexisting drift along the way, commit it separately first with a clear message (e.g. "Apply ruff format to file.py (preexisting drift)") so the task commit stays focused and the git log stays readable.

**Pre-commit hook is active.** A local `.pre-commit-config.yaml` runs `ruff check`, `ruff format --check`, `mypy --strict` and `pytest tests/unit` on every `git commit`. Activate it once after cloning with `pipeline/.venv/bin/pre-commit install`; see `docs/DEVELOPMENT.md` for the full bootstrap. Bypassing it with `--no-verify` is reserved for emergencies and must be paired with a follow-up commit that restores the gate.

## Decisioni di design rinviate

**Decisione E — Conversione `ProfilingSignals` da dataclass a Pydantic v2.** Consapevolmente rinviata. Oggi `ProfilingSignals` e i suoi otto sotto-dataclass restano `@dataclass(frozen=True)` mentre `DocumentProfile` è già Pydantic. Quando il bump additivo che emette `detection_signals` nello schema arriverà — atteso non prima della maturazione seria del profiler — `ProfilingSignals` dovrà o diventare Pydantic v2 o passare attraverso un converter dedicato. La scelta è stata rinviata per evitare di pagare il costo della conversione finché il modello non sarà esercitato da almeno un plugin di corpus reale.

## Architectural notes

**Tier 1 classification is deliberately narrow.** The generic tier 1 in `pipeline/src/scabopdf_pipeline/classification/tier1.py` only emits seven categories: `EMPTY_PAGE`, `ARTIFACT_FILIGREE`, `ARTIFACT_RUNNING_HEADER`, `ARTIFACT_FOOTER`, `BOOK_PAGE_ANCHOR`, `CROSS_REFERENCE`, `UNCLASSIFIED`. Crucially, `CROSS_REFERENCE` is recognised **only when the entire block is a single superscript span of pure digits** (heuristic `superscript_cross_reference`). Inline superscripts embedded inside larger `BODY` blocks are invisible to tier 1 and end up absorbed in `UNCLASSIFIED`. Categories like `NOTE`, `MARGINAL_HEADING`, `MARGINAL_GLOSS`, `EXAMPLE_BOX`, `INDEX_ENTRY`, `CHAPTER_SUMMARY` and the legal-code family (`ARTICLE_HEADER`, `ARTICLE_BODY`, `HEADING_N`, etc.) are **entirely profile-specific** and must be assigned by the plugin's `refine_classification` (tier 2).

**Re-parsing for inline cross-references.** A corpus plugin that needs to bind footnote superscripts to their target notes must mint synthetic `CROSS_REFERENCE` `Node` instances **in `refine_reconstruction`**, not at the `ClassifiedBlock` level. The reason is structural: `ClassifiedBlock` carries only `block_index`, `category`, `subcategory` and `reason` (no text), and the tier 1 reconstructor in `pipeline/src/scabopdf_pipeline/reconstruction/tier1.py` indexes `extraction.blocks[cb.block_index]` to materialise each Node's text, so emitting two ClassifiedBlocks pointing to the same block index would build two Nodes both carrying the full body text rather than the digit substring. The correct pattern, established by the Mosconi plugin (commit `265e6d9`), is: leave the BODY blocks as is in `refine_classification`; in `refine_reconstruction` walk every BODY Node, examine its underlying spans via `extraction.blocks[block_index].spans`, and for each inline superscript-digit span matching the profile's signature mint a synthetic CROSS_REFERENCE Node as a sibling immediately after the BODY in its parent's children list. The synthetic Node's `id` must respect the JSON schema's `^node_\d+$` pattern (Pydantic enforces it on `NodeDict.id`); mint sequentially from one past `max(int(re.match(r"^node_(\d+)$", n.id).group(1)) for n in document.root walk)`. The tier 1 generic cross-reference resolver in `pipeline/src/scabopdf_pipeline/apparatus/resolver.py` then sees the synthetic Nodes in its post-order walk and binds each to the target NOTE within the scope of the nearest HEADING ancestor. The same principle (synthetic Node minting in `refine_reconstruction`) applies to any other apparatus category whose structural unit emerges only at reading-order or extraction-span granularity, not at PyMuPDF-block granularity. The Mandrioli plugin (commit `6ee8efa`) reuses the very same minting framework — `_NodeIdMinter` seeded by `_max_existing_node_counter`, sibling-insertion after the BODY parent, identical schema-pattern compliance for the `node_NNNN` id — but feeds it from a textual regex on `node.text` rather than from an iteration over `extraction.blocks[block_index].spans`: the Mandrioli inline cross-reference markers are parenthesised digits `(N)` typeset inside ordinary 11.0pt body spans, so the typographic superscript flag is absent and the only available signal is textual. The regex `_CROSSREF_INLINE_PATTERN = re.compile(r"(?<!p\.\s)(?<!p\.)\((\d+)\)")` carries two negative look-behinds that rule out the `p. NN` page-reference pattern. This confirms that the minting framework is signal-agnostic: whether the inline marker is detected typographically (Mosconi) or textually (Mandrioli) is an orthogonal choice that the plugin makes on the basis of the corpus's own conventions.

**`ProfilePlugin` API is seven methods, not more.** The abstract base class in `pipeline/src/scabopdf_pipeline/profiling/plugin.py` exposes exactly seven abstract methods after the post-audit consolidation of 13 May 2026 (commit `615173e`): four declarative methods (`matches`, `get_categories`, `get_post_processing`, `get_layouts_disabled`) that describe the profile statically, and three tier-2 refinement hooks (`refine_classification`, `refine_reconstruction`, `refine_apparatus`) called in sequence by the orchestrator after the corresponding tier 1 phase. An earlier draft of this API exposed an additional abstract method `parse(blocks) -> Document` that was never wired into the pipeline; it has been removed and `ARCHITECTURE.md § 2.4` documents this history. New corpus plugins must subclass `ProfilePlugin` and implement all seven methods; test plugins subclass `NoOpProfilePlugin` from `pipeline/tests/conftest.py` and override only the hook they exercise. Adding a new abstract method to the ABC is a breaking change for every existing plugin and must be discussed in prose with the user first.

**Tier-2 hook signature asymmetry is deliberate.** `refine_classification` takes `(extraction, tier1_results)` while `refine_reconstruction` and `refine_apparatus` take `(document, extraction, classified_blocks)`. The first parameter differs because classification runs **before** the `Document` exists — there is nothing meaningful to pass as a first positional argument. The asymmetry is permanent and is documented in the docstring of `refine_classification` and in `ARCHITECTURE.md § 2.4`. Plugin authors should not try to "harmonise" it: passing a fake empty `Document` to `refine_classification` would be false symmetry, and rebuilding it after classification would be wasted work.

**`PageIndex` is the canonical 0-based page convention.** The type alias `PageIndex = int` in `pipeline/src/scabopdf_pipeline/extraction/types.py` marks every page-index field across Layer 1 (`Span.page`, `Block.page`, `PageGeometry.page`, `PageImageInfo.page`, `DrawingInfo.page`, `Node.page_index`, `Transformation.page_index`, `NodeDict.page_index`). The semantics matches PyMuPDF (`fitz`) exactly: the first page of the PDF is index `0`. The 1-based "book page numbers" that a manual prints on the page are **not** `PageIndex`: they live only inside `BOOK_PAGE_ANCHOR` nodes as text, and the corpus plugin that recognises them is responsible for preserving that distinction (string in the node text, not integer in a `PageIndex`-typed field). When a plugin tier 2 needs to expose a book-page mapping it should add a new dedicated field — not overload `page_index`.

The Giappichelli plugin is now consolidated on the integral four-volume Mandrioli-Carratta "Diritto processuale civile" series (Vol. I, Vol. II, Vol. III and Vol. IV of the 30th edition 2025/26) and validated against the Marotta "La cittadinanza romana" control sample, which the same editor Giappichelli publishes in a structurally different editorial pipeline (Roman-law monograph, TimesNewRomanPSMT body, Acrobat Pro 9.4.5, A4 geometry). The empirical results post-consolidation are: Vol. I (288 pages) yields 12 HEADING_2, 10 HEADING_3, 78 HEADING_4 and 18 CHAPTER_SUMMARY with no PARTE and no MARGINAL_GLOSS because Vol. I is produced through a Photoshop pipeline without AGaramondPro; Vol. II (352 pages) yields 10 HEADING_2, 15 HEADING_3, 95 HEADING_4 and 21 CHAPTER_SUMMARY; Vol. III (498 pages) yields 4 HEADING_1 (PARTE), 11 HEADING_2, 7 HEADING_3, 82 HEADING_4, 466 NOTE against the 4 pre-consolidamento, 12 MARGINAL_GLOSS, 15 CHAPTER_SUMMARY, 1531 CROSS_REFERENCE and 537 apparatus refs; Vol. IV (497 pages) yields 2 HEADING_1, 9 HEADING_2, 11 HEADING_3, 60 HEADING_4, 470 NOTE, 14 MARGINAL_GLOSS and 19 CHAPTER_SUMMARY. The Marotta control sample (206 pages) returns `matches() = 0.0`, correctly falling below the 0.6 dispatcher threshold and being routed to the `unknown_generic` fallback. Together they establish the canonical patterns for new corpus plugins: (a) typographic signatures keyed on the first span of a block, with a fallback to "any span of the right family" for heterogeneous blocks like chapter summaries; (b) profile-specific regexes for heading patterns, with explicit text-length caps to disambiguate front-matter from body headings sharing a signature; (c) instance-state warning accumulator (and chapter-pair block-index sets in the Tesauro and Mosconi cases) queued in `refine_classification` and flushed into `Document.warnings` in `refine_reconstruction`, because the classification hook has no `Document` to attach a warning to; (d) closed warning vocabulary with `plugin:<editorial_family>:` prefix (Mosconi uses `plugin:utet_wolterskluwer:` to discriminate from Tesauro's `plugin:utet:` even though they share the editorial pipeline; Mandrioli uses `plugin:giappichelli:`); (e) text parsing lives in `refine_reconstruction` operating on the already-classified tree, never duplicated in the classifier; (f) **pattern-over-signature discrimination** when the typographic difference between two categories is below PyMuPDF's measurement noise floor (Tesauro L1 vs L2 share the same body size of 10.2pt and discriminate purely on the number of dots in the numeric prefix); (g) **multi-block fusion in `refine_reconstruction`** when an editorial unit spans two consecutive blocks (Tesauro and Mosconi chapter heading: "Capitolo `<ord>`" block + title block fused into one HEADING_2 node with concatenated text and both block_indices); (h) **`matches()` symmetric discrimination via apparatus signals** for profiles that share an editorial pipeline but produce different genres (Tesauro penalises `footnote_markers >= 50` or `marginal_headings >= 50` with −0.30 to stay below the 0.6 dispatcher threshold on Mosconi; Mosconi penalises the absence of apparatus with the symmetric −0.30 to stay below the threshold on Tesauro and Patriarca; both clear the threshold only on their own corpus); (i) **generalised regex for editorial residue** (the Tesauro ARTIFACT_STAMP regex `^\d+_[A-Za-z0-9_]+\.indb?$` and `^\d{2}/\d{2}/\d{2}\s+\d{1,2}:\d{2}` covers both Tesauro and Mosconi watermarks without modification); (j) **rescue codepath from ARTIFACT_FOOTER to ARTIFACT_STAMP** for blocks that tier 1 absorbs into a broader category because of zone-y proximity, but whose text signature pinpoints a finer-grained kind. The Mosconi plugin adds three new structural patterns: (k) **synthetic Node minting in `refine_reconstruction` for tier 2 categories that emerge inline inside tier 1 blocks**, with `id` minted following the `node_NNNN` schema pattern starting one past the highest counter already assigned by tier 1 (Mosconi inline CROSS_REFERENCE superscripts); (l) **two-pass classification with deferred title promotion** when two structurally distinct categories share the same typographic signature and the discriminator is ordering-based (Mosconi HEADING_1 front-matter heading vs HEADING_2 chapter title: the first pass leaves every 12pt non-chapter-number block provisional as HEADING_1, the second pass `_register_chapter_pairs` promotes the next non-stamp HEADING_1 after each chapter-number to HEADING_2 chapter title, tolerating intervening ARTIFACT_STAMP blocks); (m) **textual marker pattern verified against the full block text, not the leading span**, when the marker glyph (e.g. the small footnote number) lives in one span and the rest of the marker (the dot, the whitespace) lives in the following span at a different size — the predicate combines a typographic check on the leading span with a textual check on `view.text.lstrip()` rather than `view.spans[0].text`. The Mandrioli plugin adds ten more structural patterns, five from the original commit `6ee8efa` and five from the consolidation commit `c01661e`: (n) **CROSS_REFERENCE minting driven by a textual regex on `node.text`** rather than by typographic superscript flags on `extraction.blocks[bi].spans` — Mandrioli's inline markers are parenthesised digits `(N)` typeset at the same 11.0pt as the surrounding body, so the only available signal is textual; the same `_NodeIdMinter` and sibling-insertion machinery that Mosconi established is reused unchanged, with `_CROSSREF_INLINE_PATTERN = re.compile(r"(?<!p\.\s)(?<!p\.)\((\d+)\)")` (the two negative look-behinds rule out the `p. NN` page-reference pattern). (o) **Small-caps three-span pattern as a category discriminator**, when the typesetter renders a heading or label as `(leading-letter at nominal-size, middle-letters at smaller-size, trailing-tail at nominal-size)` rather than as a homogeneous-size unit: Mandrioli uses this for SOMMARIO (CHAPTER_SUMMARY label, otherwise confusable with NOTE because both share the 9.0pt body size) and for CAPITOLO (otherwise indistinguishable from a generic 13.0pt block), and the predicate must therefore check `spans[0].size`, `spans[1].size` and `spans[1].text.startswith("OMMARIO" or "APITOLO")` jointly, never the block's average size. (p) **Body+note glued-block diagnostic warning**, for the case in which PyMuPDF fuses a body block (11.0pt) and the immediately following note block (9.0pt or 7.98pt depending on the editorial pipeline) into a single block: on the Mandrioli Vol. III fixture this affects roughly two thousand blocks (well above the under-one-percent estimate from a five-page empirical sample), and the plugin detects the pattern with the predicate "BODY block whose fraction of NOTE-sized spans exceeds 30 %" and emits the closed warning `plugin:giappichelli:body_note_block_glued_block_<idx>_page_<p>` at classification time. This warning coexists with the structural splitter described in pattern (u): the upstream warning marks every glued block as detected, the splitter then materialises one synthetic NOTE Node per transition and emits its own per-Node warning `plugin:giappichelli:body_note_split_minted_node_<id>_page_<p>`. The delta between the two counts is a diagnostic gauge for "glued blocks the splitter could not recover". (q) **HEADING_4 in production for the first time** — Mandrioli's paragrafo numerato has a two-span composite typographic signature `[Roman 11.5pt number] [Italic 11.5pt title]` with no structural bold (the Giappichelli pipeline expresses emphasis via italic and maiuscoletto rather than bold), so the HEADING_4 predicate must look for italic on the title span rather than the bold flag on which the prior three plugins rely. (r) **MARGINAL_GLOSS filled in production for the first time** — Mosconi exercised MARGINAL_HEADING (a different category), and Mandrioli is the first plugin to populate MARGINAL_GLOSS, with both `AGaramondPro-BoldItalic` and the subset-truncated `AGaramondPro-SemiboldIta` (PyMuPDF truncates the canonical `AGaramondPro-SemiboldItalic` to 23 characters in some output) admitted as valid font fragments; the geometric guard `bbox.x0 < 50` excludes the body-column fallback where the same font appears inside the body column (e.g. the p.170 "cu-/ratore del minore" InDesign glyph substitution). (s) **Five-span small-caps composite as a category discriminator for top-level divisions**, when the typesetter renders the highest division of the book (PARTE, in the Mandrioli Vol. III and Vol. IV) as a five-span small-caps composite of the shape `[("P", 13.98pt) + ("ARTE ", 10.98pt) + ("P", 13.98pt) + ("RIMA", 10.98pt) + (..., 13.98pt)]` whose joined text reads "PARTE PRIMA". The plugin's `_is_parte_signature` predicate verifies the 13.98pt + 10.98pt size pair on the first two spans and accepts any block with at least four spans following that pattern; the predicate is dispatched upstream of `_is_chapter_heading_signature` (CAPITOLO at 13.02pt) so that PARTE is recognised before the chapter-heading branch can consume it. (t) **Dual-regime size detection for the same category across editorial pipelines**, when the same publisher produces a series whose volumes traverse two different typesetting pipelines that yield two different nominal sizes for the same logical category: the Mandrioli NOTE exists in two regimes, 9.0pt on the InDesign-derived Vol. III and Vol. IV, 7.98pt on the Photoshop-derived Vol. I and Vol. II, and the plugin's `NOTE_BODY_SIZES = (7.98, 9.0)` closed tuple lets the `_is_note` predicate match either; the body size (10.98pt) is uniform across the four volumes and only the NOTE has the dual regime. (u) **Body+note glued blocks splitter mints synthetic NOTE Nodes in `refine_reconstruction`**, the structural counterpart of the diagnostic warning of pattern (p): PyMuPDF occasionally fuses a body block and the immediately following note block into a single block (this affects roughly 95 % of Vol. III blocks, 6.96 % of Vol. IV, and approximately zero on Vol. I and Vol. II because the Photoshop pipeline emits cleaner block boundaries), and the plugin's `_split_body_note_glued` walks every BODY Node carrying a single block_index, recovers the original spans via `extraction.spans[start:end]`, verifies the glued predicate (≥30 % of spans at NOTE_BODY_SIZES), finds every transition point (a NOTE-sized span whose text opens with the `(N)` marker), truncates the BODY text to the spans before the first transition, and mints one synthetic NOTE Node per transition as a sibling immediately after the truncated BODY in the parent's children list. The synthetic NOTE Nodes reuse the same `_NodeIdMinter` consumed downstream by the CROSS_REFERENCE minting machinery, inherit the source block_index, and each one emits a per-Node warning `plugin:giappichelli:body_note_split_minted_node_<id>_page_<p>`. The empirical impact on Vol. III is a 115x improvement from n_note=4 to n_note=466. (v) **Three limitations of the `Transformation` model on structural splits**, documented here as a forward-looking note for the future schema bump that will close them. When a corpus plugin performs a structural transformation that decomposes a Node into multiple Nodes — the Mandrioli body+note splitter that splits one BODY into BODY + N synthetic NOTE siblings (commit `c01661e`), or the Mosconi `recompose_marginal_ellipsis` that consolidates several MARGINAL_HEADING siblings into one (commit `265e6d9`) — the current `Transformation` model in `postprocessing/types.py` does **not** record the materialisation of synthetic Nodes or the absorption of consumed Nodes, only the textual intra-Node substitutions on the surviving Node. The three consequences worth documenting for future maintainers are: first, the decomposition or merge is a structural tree transformation and not a textual intra-Node substitution; second, the model preserves the textual reversibility of the surviving Node but not the implicit materialisation or absorption of siblings; third, an additive 0.5.0 schema bump that extends `Transformation` with optional `split_into` and `merged_from` fields (lists of node ids generated by, or absorbed into, the operation) would close the gap and must be planned in a dedicated session, not retrofitted under the pressure of a corpus-plugin session. Two real plugins now exhibit this limitation in production (Mandrioli body+note splitter, Mosconi `recompose_marginal_ellipsis`), and three additional plugins (BIC `_split_body_note_multi_block`, DeJure NS notes-section consolidator, DeJure Dottrina notes-section consolidator) now leverage the dual-field reversibility added in the 0.5.0 bump, which makes the original bump a concrete need that has been fulfilled rather than a hypothetical one. (w) **Regression-protection integration test for `matches()` non-promotion on a sibling-editor non-target document**, established at the consolidation of the Giappichelli plugin: the plugin is calibrated on the Mandrioli-Carratta series, and Marotta "La cittadinanza romana" — published by the same editor Giappichelli but editorially a different book (Roman-law monograph, TimesNewRomanPSMT body, Acrobat Pro 9.4.5 pipeline, A4 geometry) — is the natural false-positive risk. The `test_manuale_giappichelli_does_not_promote_on_marotta_fixture` integration test builds `ProfilingSignals` from the real Marotta fixture and asserts that `matches()` returns strictly below the 0.6 dispatcher threshold (empirically clamped to 0.0 by the family penalty); its companion test verifies that the dispatcher fallback `UnknownGenericProfile` processes Marotta with a flat tree of depth 1. The two tests catch any future loosening of `matches()` that would extend the plugin's scope beyond Mandrioli, and the same pattern should be replicated by future plugins whenever the editor publishes structurally heterogeneous books under the same family. The BIC plugin (`manuale_bic`, Marrone "Istituzioni di Diritto Romano" BIC 2009, commit `<TBD>` for the v1 landing plus `<TBD>` for the immediate consolidation that closed the three v1 residues on the same day's later night session) adds three more structural patterns to the corpus-plugin convention, all driven by the empirical inspection of the integrale 684-page fixture: the Marrone is typeset on A4 in Verdana monocromatico (a single body family used at multiple sizes) and exploits four typographic **colours** — green for HEADING_1, indigo for HEADING_2, maroon for HEADING_3, red for the in-line Note marker — as its primary structural discriminator, a registration regime that the prior five plugins did not exercise because their corpora carry the structural signal in font, size or weight rather than colour. The empirical post-consolidation post-classification structure is 13 HEADING_1 (the green-coloured chapter heads), 1 HEADING_2 (the indigo "Premesse" front-matter heading), 208 HEADING_3 (maroon `§ <n>` paragrafi), 2261 BODY, **1454 NOTE** (individual notes recovered by the body+note splitter described in pattern (kk) plus the note-continuation rescuer described in pattern (mm) — was 1261 in the v1 landing), **1489 CROSS_REFERENCE** (was 1548 in the v1 landing; the post-consolidation drop comes from the back-matter filter that skips minting under Bibliografia/Indice analitico HEADING_1 because their numeric markers point to bibliographic entries, not NOTE targets), **1473 BOOK_PAGE_ANCHOR** (was 594 in the v1 landing; the consolidation walker described in pattern (nn) mints synthetic anchors for every 0.96pt Verdana/Arial numeric span inside mixed blocks, on top of the 665 all-1pt blocks still handled by the tier 1 `tiny_font_anchor`), 693 ARTIFACT_FOOTER and 5 ARTIFACT_STAMP (the volume frontispieces). The plugin's `refine_apparatus` overrides the tier 1 generic resolver with a per-chapter forward-scan binding described in pattern (ll) — the tier 1 generic CROSS_REF_TARGET resolver scans backwards from the CROSS_REFERENCE looking for the NOTE within the nearest HEADING ancestor, but the Marrone editorial convention is the opposite (cross-references in the body PRECEDE the Note section of the same paragrafo), so the plugin walks pre-order DFS tracking the `current_chapter` (last HEADING_1 root-level), builds a per-chapter `marker → note_node_id` index forward, and binds **1430 of 1489 CROSS_REFERENCE (96.0%)** to their target NOTE — well above the 64% achieved by the tier 1 generic resolver before the override. No schema bump was needed: the `BOOK_PAGE_ANCHOR` category and the `Span.color` field were already present at schema 0.5.0, so the colour-driven discrimination plugs into the existing contract surface without touching `contract.py` or `shared/schema.json`. The three new patterns are: (ii) **tier 1 ARTIFACT_RUNNING_HEADER → HEADING_N rescue**, when the editorial pipeline typesets the highest-level headings at the very top of the page (inside tier 1's header-zone heuristic boundary) and the generic classifier therefore absorbs them as `ARTIFACT_RUNNING_HEADER` before any plugin predicate can run. The Marrone places its 13 green chapter heads in the page header zone, and the v1 implementation of tier 2 that applied the colour-specific HEADING predicates only to UNCLASSIFIED tier 1 verdicts mis-classified 12 of the 13 chapter heads as `ARTIFACT_RUNNING_HEADER` (a single chapter head landed below the header threshold and was caught). The plugin's `_heading_rescue` applies the colour-specific HEADING_1/2/3 predicates to **all** tier 1 verdicts (not just UNCLASSIFIED) and reclassifies on positive match; the rescue is dispatched before any other tier 2 predicate so that the same block cannot be subsequently absorbed by a downstream predicate. Future plugins observing a structural category that tier 1 systematically buries in `ARTIFACT_RUNNING_HEADER` or `ARTIFACT_FOOTER` should replicate this rescue pattern rather than retighten the tier 1 zone boundaries, which would risk false negatives on documents that conform to the default header zone. (jj) **permissive any-span BODY predicate**, when the corpus interleaves small inline spans (cross-references, anchors, emphasis) at the leading position of body blocks and a leading-span body check therefore fails on a significant fraction of legitimate body blocks. The Marrone places its 0.96pt Verdana book-page anchors and its 10.56pt Verdana cross-reference superscripts as the **first** span of many body blocks (because the anchor / superscript renders in the page-margin column that PyMuPDF reads before the body column on the same line, or because the superscript opens the paragraph after a heading), defeating the leading-span body check that the prior five plugins relied on. The plugin's `_is_body` predicate accepts every block where **at least one** span has the body signature (Verdana 12pt regular), iterating through the spans rather than inspecting only the first; HEADING predicates are dispatched **before** `_is_body` so that a genuine heading whose first span is the title text and whose remaining spans are body-sized is not accidentally promoted to BODY. Future plugins observing similar interleaved small-glyph leading spans should reuse the any-span permissive body check combined with strict before-body dispatch ordering of the HEADING predicates, rather than encoding a complex leading-span exclusion list. (kk) **multi-block body+note splitter with line-level marker detection**, when the editorial pipeline groups notes per paragrafo with an in-line "Note" marker (Verdana,Bold 12pt colour `#ff0000`) and PyMuPDF fuses body + marker + notes into a single block, sometimes spanning multiple block_indices via tier 1 cross-page paragraph merging. On the Marrone fixture roughly 60 % of the body+note cases produce a multi-block BODY with the marker landing in one of the block_indices rather than the first; the Mandrioli body+note splitter, which assumes a single block_index per glued BODY, would miss every multi-block case. The plugin's `_split_body_note_multi_block` walks every block_index attached to the BODY Node (not just the first), finds the marker block by scanning each block's spans for the red Verdana,Bold 12pt signature at the **line** level (a marker line is one whose effective leading span carries the marker signature, regardless of where it sits in the block-global span sequence), splits at the marker line, keeps the prefix block-indices and prefix spans as the surviving BODY's text, and mints one synthetic NOTE Node per `N. ` transition in the post-marker spans. The transformation populates `Transformation.split_into` with the minted ids per schema 0.5.0, recording the structural decomposition reversibly in the audit log. The pattern reuses the Mandrioli body+note splitter framework — the same `_NodeIdMinter` seeded by `_max_existing_node_counter`, the same sibling-insertion of synthetic NOTE Nodes immediately after the truncated BODY, the same per-Node `plugin:bic:body_note_split_minted_node_<id>_page_<p>` warning vocabulary — but the marker signal is **line-level colour-specific** (the red Verdana,Bold 12pt line) rather than the **span-level size signature** (NOTE_BODY_SIZES at 7.98pt or 9.0pt) that Mandrioli uses; this confirms once more that the body+note splitting framework is signal-agnostic and that the choice between line-level and span-level detection is an orthogonal decision the plugin makes on the basis of the corpus's typographic conventions. The immediate consolidation session that landed the same night (closing the three v1 residues dichiarated in CARRYOVER v2.8) adds three further structural patterns: (ll) **Cross-reference forward scan per per-chapter scope override**, when the editorial convention places the inline cross-reference markers in the body *before* the Note section of the same paragrafo rather than *after* it (Marrone, BIC manuals, plausibly other accessible didactic manuals). The tier 1 generic `_resolve_cross_references` resolver in `apparatus/resolver.py` scans **backwards** from the CROSS_REFERENCE Node looking for the NOTE within the nearest HEADING ancestor, which on a corpus with the Marrone convention either fails universally (no NOTE precedes the CR within the scope) or binds to the wrong NOTE of a previous chapter. The plugin must override the binding logic in `refine_apparatus` with a pre-order DFS walk that tracks the `current_chapter` (last HEADING_1 root-level), builds a per-chapter `marker → note_node_id` index forward, and binds each plugin-minted CROSS_REFERENCE to the NOTE that follows it in pre-order order within the same chapter scope. Importantly, even when the `§ paragrafi` HEADING_3 are root-level orphans (the tier 1c hierarchy assembler places HEADING_3 at root when the heading_stack has no HEADING_2 between HEADING_1 and HEADING_3), the chapter tracking survives because it follows HEADING_1 boundaries rather than subtree ancestry. The pattern also filters the tier 1 `unresolved_cross_reference_node_<id>_n_<N>` and `unparseable_cross_reference_node_<id>` warnings for every plugin-minted CR that became redundant after the override; the override emits its own `plugin:bic:cross_reference_unresolved_node_<id>_marker_<N>` for the CR that have no matching NOTE within the chapter scope. A companion editorial-pattern guard skips cross-reference minting under HEADING_1 marked as back-matter (Bibliografia, Indice analitico, matched by `_BACK_MATTER_HEADING_PATTERN`) because the numeric markers there point to bibliographic entries, not NOTE targets — the guard avoids polluting the binding-rate denominator with unbindable CR. (mm) **Note continuation rescue with structural guard "preceding-content-is-NOTE"**, when the tier 1 cross-page paragraph merger refuses to fuse a BODY top-of-page into the preceding BODY because the second block's text matches a heading pattern (on the Marrone, `"13. New note..."` matches the generic `detect_heading_pattern` numbered-paragraph regex, short-circuiting the merge). The plugin must recognise the standalone BODY as a note continuation structurally. The most robust guard is **not** "the first span is size N" or "the text matches pattern X" — both also capture genuine body paragraphs that happen to open with `^N. `. The right guard is **"the preceding meaningful sibling (skipping ARTIFACT_FOOTER, BOOK_PAGE_ANCHOR, ARTIFACT_RUNNING_HEADER, ARTIFACT_FILIGREE, ARTIFACT_STAMP, EMPTY_PAGE) is a NOTE Node"**. The preceding-sibling skip is necessary because the tier 1c reading-order sort interleaves page footers and book-page anchors between the last NOTE of page N and the first content of page N+1, so the immediate preceding sibling is often an ARTIFACT, not the NOTE itself. Combined with a magnitude check on the marker number (`≤ NOTE_MARKER_MAX_VALUE = 500`, ruling out year references and other large numbers that are not note markers), the guard reduces false-positive promotions to zero and the per-chapter marker index stays clean. The rescued BODY is converted into one or more synthetic NOTE Nodes via `_convert_body_to_notes`, which reuses the `_group_note_spans` framework already established by the body+note splitter; per-Node warning `plugin:bic:note_continuation_rescued_node_<id>_page_<p>_marker_<N>` is emitted on success (with a fallback variant without the `_marker_<N>` suffix if the grouping fails for a particular block). (nn) **BOOK_PAGE_ANCHOR inline minting for mixed blocks**, when the editorial pipeline (BIC accessibility) embeds 0.96pt invisible page anchors in the left margin fused with 12pt body spans inside single blocks (mixed blocks), defeating the tier 1 generic `tiny_font_anchor` heuristic which requires ALL spans of the block to be below 2pt. On the Marrone fixture roughly 670 anchors are inline-fused this way and roughly 665 stand alone in all-1pt blocks; the v1 plugin recovered only the latter via tier 1, leaving the former invisible to Layer 2. The plugin walks every Node carrying `block_indices`, inspects each span of each block, and mints a synthetic BOOK_PAGE_ANCHOR Node as a sibling after the host Node for every 0.96pt Verdana/Arial numeric span outside all-1pt blocks. The pattern is signal-agnostic: future plugins for editorial pipelines with the same inline-anchor pattern (Giuffrè penal/civil codes with page anchors, historic EdD with OCR anchors, Zanichelli manuals with sub-text anchors) can reuse the same `_mint_anchors_for_node` framework with thresholds adapted to their corpus.

**Warning framework consolidation.** Landed on 2026-05-21 (Fase 2 of the Ambitious Plan, P-007 + P-032) as the operational unification of the closed warning vocabulary previously distributed across the 13 corpus plugins, the tier 1 generic emitters, and the hand-curated test registry. **Opt-in ABC extension via non-abstract classmethod.** Landed on 2026-05-21 (Fase 2, same session as pattern (sss)) as the architectural convention for additive ABC extensions that must not break the "stable abstract surface" invariant. The new pattern (ttt) **Opt-in ABC extension via non-abstract classmethod** documents the project's policy: when an ABC's abstract surface has been declared stable at a specific count (the `ProfilePlugin` ABC declared at "stable at 7 abstract methods" since the audit consolidation of 13 May 2026 commit `615173e`), and a new capability is required that subclasses may or may not override, the new capability must be exposed as a **non-abstract classmethod with a sensible default** rather than as a new abstract method. The Fase 2 example added `classmethod get_warning_templates(cls) -> tuple[str, ...]` with default `return ()` on `ProfilePlugin`; the ABC's count of *abstract* methods stays at seven (the new classmethod is additive and non-binding), every concrete plugin that does override declares its closed warning vocabulary via a four-line `return WARNING_TEMPLATES`, and the fallback `UnknownGenericProfile` inherits the empty default because it emits no warnings. Two alternatives were considered and rejected during Fase 1's autonomous decision: (a) a global module-level registry where each plugin module calls `register_warning_templates(MyPlugin, WARNING_TEMPLATES)` was rejected as side-effect-y and magical (the registration depends on import order and the side-effect is invisible at the plugin class definition); (b) discovery via `__dict__` introspection where the framework reads `getattr(sys.modules[plugin_class.__module__], "WARNING_TEMPLATES", ())` was rejected as fragile (depends on the module-level constant name being exactly `WARNING_TEMPLATES`) and magical (no API surface). The non-abstract classmethod is the type-safe, discoverable, opt-in choice. The pattern generalises to any future ABC extension where a new capability is "nice to have on most subclasses but not required by the abstract contract": add a non-abstract classmethod or method with the sensible default, document it in the ABC docstring, override on concrete subclasses that want non-default behaviour. The pattern preserves the "stable at N abstract methods" invariant while opening the ABC to evolution.

The DeJure NS is the **first plugin operating on the Aspose.PDF for .NET editorial pipeline** (Aspose 18.4, Arial family in `ArialMT`/`Arial-BoldMT`/`Arial-ItalicMT`/`ArialItalic` variants, Letter 612 x 792pt geometry), distinct from the prior six pipelines (PDFsharp, InDesign, iLovePDF, Photoshop, Acrobat Pro, accessibility BIC), and is also the first plugin to exercise the **bimodal archetype continuum** under one banner `NOTE E DOTTRINA`: the short narrative archetype carries no sommario, no numbered sections and no notes section, opening with an editorial `SUBTITLE` line `"Quotidiano del <date>"` between metadata and body, while the long academic archetype opens with a `TOC_GENERAL` prose sommario, several uppercase numbered `HEADING_1` sections (5 on the long fixture), a closing `Note:` `SECTION_LABEL` and a single concatenated block of 54 numbered `(N)` notes that the plugin splits into individual NOTE Nodes. The plugin lights up for the first time in production a previously-dormant slice of the schema 0.5.0 `SemanticCategory` enum — `GENRE_BANNER`, `TITLE`, `FONTE_VALUE`, `REFERRAL`, `AUTHORS`, `META_VALUE`, `META_LABEL`, `SUBTITLE`, `SECTION_LABEL`, `TOC_GENERAL` — validating both the upstream schema choice and the conversion convention at the emission boundary without any schema bump. The empirical post-classification structure on the long fixture is 1 GENRE_BANNER, 1 TITLE, 1 META_VALUE umbrella decomposed into 3 sibling Nodes (FONTE_VALUE + REFERRAL + AUTHORS), 1 TOC_GENERAL with 5 parsed `toc_items`, 5 HEADING_1 numbered sections, 1 SECTION_LABEL `Note:`, **54 synthetic NOTE Nodes** minted from the post-`Note:` notes section block, **53 CROSS_REFERENCE Nodes** minted from inline `(N)` markers in body text and bound at **100 % rate** to their target NOTE via the global per-document marker → node_id index; 22 ARTIFACT_FOOTER (`Pagina N di M` italic) and 1 ARTIFACT_STAMP (`SERVIZIO GESTIONE RISORSE DOCUMENTARIE...` copyright). The plugin's closed warning vocabulary, prefix `plugin:dejure_nota_sentenza:`, lists 8 templates in `WARNING_TEMPLATES`: `metadata_block_unparseable_block_<idx>_page_<p>`, `metadata_field_minted_node_<id>_field_<name>`, `toc_general_parsed_node_<id>_items_<n>`, `toc_general_unparseable_node_<id>`, `section_heading_pattern_unmatched_block_<idx>_page_<p>`, `note_section_split_minted_node_<id>_page_<p>_marker_<n>`, `cross_reference_minted_node_<id>_page_<p>_marker_<n>`, `cross_reference_unresolved_node_<id>_marker_<n>`. The three new patterns are: (oo) **Metadata block decomposition via regex split on label markers**, when the editorial pipeline emits a metadata block whose internal label/value lines are concatenated by the PDF renderer into a single span sequence without newline separators (the Aspose.PDF for .NET pipeline emits the DeJure NS three-line metadata block `"Fonte: ...Nota a: ...Autori: ..."` as one contiguous string per block, with the line discriminator embedded only in PyMuPDF's `line_index` metadata which is lost at the Node-text level). A `str.splitlines()` decomposer yields a single chunk and the plugin would fail to surface the three structured metadata Nodes. The correct pattern is a positive look-ahead regex split on the closed set of label markers (`re.split(r"(?=Fonte:|Nota a:|Autori:)", host.text)`) — each chunk then starts with its own label and can be matched against the closed set to identify the field. The decomposition is recorded as a `Transformation` with `split_into` populated per the schema 0.5.0 structural reversibility convention; the host umbrella Node is removed from its parent's children list and replaced by the three synthetic field Nodes (FONTE_VALUE, REFERRAL, AUTHORS for DeJure NS). The pattern is signal-agnostic and applies to any future plugin whose editorial pipeline emits multi-field blocks with concatenated lines, identified by a closed set of textual label markers. The plugin emits a per-occurrence `metadata_block_unparseable_*` warning when any of the expected labels is absent. (pp) **Stateful notes-region retagging in `refine_classification`**, when the tier 1 cross-page paragraph merger would erroneously fuse the last body paragraph of a section with the first BODY block of the notes-section continuation on the next page. The DeJure NS body and notes-section continuation share the same BODY classification at tier 1 default, and the tier 1 reconstructor's merger sees BODY → SECTION_LABEL → ARTIFACT_FOOTER → BODY in reading order across the page break and would merge the two BODY Nodes into one mis-classified composite (the body of section 5 followed by the notes prose continuation pages). The plugin must run a stateful second pass over the tier 1 verdicts: walk in reading-order block_index order, when a SECTION_LABEL `"Note:"` block is encountered set an `in_notes_region` flag, and from that point onwards reclassify every BODY verdict to NOTE so the cross-page paragraph merger sees BODY → SECTION_LABEL → ARTIFACT_FOOTER → NOTE and does not merge across the BODY/NOTE category boundary. The flag persists to the end of the document because the notes section runs through to the closing ARTIFACT_STAMP. The pattern is structurally robust to other corpora whose pipelines emit a "stop merging BODY here" boundary marker (Massime label, Dottrina banner, end-of-section indicators) that the tier 1 reconstructor does not honour without per-plugin guidance. (qq) **Multi-sibling notes section consolidation with passthrough of interleaved artifacts**, the structural counterpart to pattern (pp) running in `refine_reconstruction`. The notes section in DeJure NS opens with a SECTION_LABEL `"Note:"` Node and is followed by a stream of NOTE-or-BODY Node siblings (after pattern (pp)'s retagging) interleaved with ARTIFACT_FOOTER Nodes on each page boundary. The consolidator walks the children list of the parent (typically the last HEADING_1 of the document) looking for the SECTION_LABEL marker; when found, it walks forward absorbing every BODY/NOTE sibling into a single concatenated text and passing through ARTIFACT_FOOTER (and similar non-structural artifacts: ARTIFACT_RUNNING_HEADER, ARTIFACT_PAGE_HEADER, ARTIFACT_STAMP, ARTIFACT_FILIGREE, BOOK_PAGE_ANCHOR, EMPTY_PAGE, UNCLASSIFIED) by keeping them in reading order after the synthetic NOTE Nodes; the absorption stops at the next structural boundary (HEADING_N, SECTION_LABEL, TITLE, GENRE_BANNER, META_VALUE, FONTE_VALUE, REFERRAL, AUTHORS, SUBTITLE, TOC_GENERAL) or end of the children list. The consolidated text is split on the regex `(?=\(\d+\)\s)` and one synthetic `NOTE` Node is minted per `(N) ...` chunk, with the verbatim marker preserved on the Node's `text` for the apparatus binding pass to match against. A single `Transformation` with both `split_into` (the minted NOTE ids) and `merged_from` (the absorbed sibling ids) is recorded per consolidation, leveraging the schema 0.5.0 dual-field reversibility that no prior plugin exercised simultaneously. The pattern generalises to any corpus whose notes section runs as a multi-sibling sequence with editorial artifacts interleaved.

**Post-processing step `recompose_marginal_ellipsis`.** Promoted from placeholder to real implementation in commit `265e6d9` together with the Mosconi plugin. The step lives in `pipeline/src/scabopdf_pipeline/postprocessing/steps/recompose_marginal_ellipsis.py` and fuses chains of `MARGINAL_HEADING` nodes split across two or more consecutive segments by a typographic `...` continuation marker. Crucially, the step walks the document tree in **DFS pre-order across the whole forest**, not within siblings: a chain's head segment and its terminal segment can sit under different parents (typically two consecutive HEADING_3 paragraphs straddling the page break that broke the marginal phrase). The merged head retains its original tree position; absorbed segments are removed from their (possibly different) parents. A single `Transformation` is recorded per chain on the head node, capturing the intra-Node substitution of the trailing marker with the appended content of the absorbed segments. The textual reversibility convention of `postprocessing.types.Transformation` is preserved (head text is byte-for-byte recoverable from the log), but the structural reversibility of the absorbed-segment deletion is **not** encodable in the current model; a future schema bump extending `Transformation` with optional `merged_from` fields could provide it. The trailing/leading marker regex accepts both the ASCII `...` and the Unicode horizontal ellipsis U+2026, because PyMuPDF emits both depending on the font and the source text layer. Plugins for other corpora that exhibit similar typographic continuation can declare this step in `get_post_processing()` and reuse the same implementation; the step is corpus-agnostic.

**`merge_cross_page_notes` was promoted from placeholder to real implementation at schema 0.5.0.** The closure session of 18 May 2026 (CARRYOVER v2.6) lands the step as `pipeline/src/scabopdf_pipeline/postprocessing/steps/merge_cross_page_notes.py`, registered as a real callable in `PostProcessingRegistry.default()`. The plugin `manuale_giappichelli` is the first (and currently only) declarer in `get_post_processing()`, returning `["dehyphenate_with_log", "merge_cross_page_notes"]`. The tier 1 generic resolver `_resolve_cross_page_note_merging` in `apparatus/resolver.py` now skips its own pass on documents whose plugin declares the step (the conditional check is on `plugin.get_post_processing()`), avoiding double-merging; the three prior plugins (Patriarca, Tesauro, Mosconi) do not declare the step and keep their cross-page merging at tier 1 without a Transformation log.

The promotion is accompanied by three structurally distinct patterns that future plugins should reuse: (x) **multi-note splitter pattern** — the Giappichelli `_split_body_note_glued` was extended to process both BODY and NOTE blocks. For BODY blocks (glued body+note), the surviving BODY keeps the spans before the first transition and synthetic NOTE siblings are minted for each transition. For NOTE blocks (multi-note glued where PyMuPDF emits N consecutive footnotes as one block), the surviving NOTE keeps the spans from the first transition up to the second, and synthetic NOTE siblings are minted for each additional transition. Single-transition NOTE blocks (regular single-note blocks) are left untouched. The pattern handles both editorial cases via one code path with category-based branching. (y) **line-index + span-index discriminator for fresh-note vs inline cross-reference** — `_find_note_transitions` now checks both `span.line_index != prev_note_line_index` AND `span.span_index == 0` before treating a marker-bearing 9pt span as a transition. The two-guard combination filters out inline cross-references that happen to land at the start of a wrapped line inside another note's text. PyMuPDF's `span_index` is per-line (not block-global), so `span_index == 0` means "first span of its line"; the line-index check disambiguates same-line subsequent markers. (z) **effective leading span pattern** — `_effective_leading_span(spans)` returns the first non-whitespace span instead of the raw first span, absorbing the leading whitespace fallback spans (Courier, Mangal, Cambria at 7.98pt) that PyMuPDF emits at the start of ~300 Vol. III footnote blocks. The pattern is consulted by both `_is_note` (classification predicate) and `_is_glued_spans` (splitter predicate); future plugins observing similar fallback artefacts can reuse the same `_effective_leading_span` static helper.

A consequential structural pattern emerged on Vol. I and Vol. II: (aa) **dual-family marginal annotation classification**. Vol. III/IV typeset marginal annotations in AGaramondPro 8.52pt as `MARGINAL_GLOSS` (intercepted by `_is_marginal_gloss`); Vol. I/II of the same series typeset them in SimonciniGaramondStd 7.98pt as `MARGINAL_HEADING` (a different semantic category) sitting in either the left margin (`bbox.x1 < 110pt`) or the right margin (`bbox.x0 > 370pt`). The new `_is_marginal_heading` predicate intercepts these Vol. I/II blocks before `_is_note` so the 7.98pt note signature does not absorb them. Documenting this here as a forward-looking pattern: a future plugin observing the same dual-family pattern (one volume of a series uses font A for marginals, another volume uses font B) should follow the same predicate-ordering convention — intercept the secondary family first, fall through to the primary `_is_note` for actual notes.

The Giuffrè diretto plugin (`manuale_giuffre_diretto`, Torrente-Schlesinger 25ª ed., commit `022357a`) adds seven more structural patterns to the corpus-plugin convention, all driven by the empirical inspection of the integrale 1559-page fixture: (bb) **four-level heading hierarchy with mixed signature/text-pattern discrimination**, the first plugin in the project that emits HEADING_1..4 densely populated (13 PARTE + 4 front/back-matter HEADING_1 = 17 total, 82 CAPITOLO HEADING_2, 56 SOTTO-SEZIONE HEADING_3, 710 § PARAGRAFO HEADING_4) within a single corpus. The four ranks combine three distinct discrimination strategies: HEADING_1 PARTE uses a pure typographic signature (TimesNewRomanPS-BoldMT 12.97pt, unique to the category); HEADING_2 CAPITOLO reuses Mandrioli's small-caps three-span pattern but with a different family pair (MScotchRoman 11.47pt + 7.73pt + 11.47pt instead of SimonciniGaramondStd 13.02pt + 10.5pt + 13.02pt); HEADING_3 SOTTO-SEZIONE has no unique typographic signature (it shares the body MScotchRoman 11.47pt regular) and discriminates purely on text pattern + bbox centering (see pattern (ff) below); HEADING_4 § PARAGRAFO uses a unique 10.98pt § glyph leading span. The pattern documents that a single plugin can legitimately combine all three discrimination strategies across its heading levels; future plugins should not feel forced to pick a single strategy per plugin. (cc) **Multi-subtype CROSS_REFERENCE minting via multiple regex patterns in sequence**, when a corpus expresses several semantically distinct kinds of inline reference (Torrente has three: `§ N` paragraph rinvio, `art. N c.c.` codice civile article, `Cass. <data> n. <n>` Court of Cassation sentence). The plugin's `_mint_for_body` runs the three regex patterns over the BODY text, sorts the matches by textual occurrence, and mints one synthetic CROSS_REFERENCE Node per match with the verbatim marker text preserved on `Node.text`. The subtype is encoded in two places: the leading character of `Node.text` (`§` vs `art.` vs `Cass.`), and the per-mint warning prefix (`plugin:giuffre_diretto:cross_reference_paragraph_minted_*` vs `_article_minted_*` vs `_sentence_minted_*`). Layer 2 can distinguish the three sub-types without a schema-level `subtype` field; a future 0.6.0 bump can promote it to a real field if production exposure shows the need. (dd) **Global-document scope for CROSS_REFERENCE binding**, when the corpus numbers its referent targets continuatively across the whole document rather than per-chapter (Torrente paragrafi 1…693 plus 17 `-bis`/`-ter` are continuative across the entire 1559-page manual). The tier 1 generic `_resolve_cross_references` resolver scopes to the nearest HEADING_1/HEADING_2 ancestor by design, which would systematically fail on this corpus. The plugin's `refine_apparatus` overrides via: (1) build a global `marker → node_id` index by walking every HEADING_4 in the document tree with `_PARAGRAFO_TITLE_MARKER_PATTERN`, normalising the marker through `_normalise_marker` (`130-bis`/`130 bis`/`130bis` all map to canonical `130-bis`); (2) walk every synthetic CROSS_REFERENCE Node minted by `refine_reconstruction` (tracked in `_minted_crossref_ids` instance-state across the two hooks); (3) for each `§ N` node, look up the marker in the index and attach an `ApparatusRef(CROSS_REF_TARGET, target_node_id, source_marker="§ N")`. `art.` and `Cass.` external references stay with empty `apparatus_refs` because they have no internal target — Layer 2 distinguishes them by inspecting `apparatus_refs` presence. (ee) **Tier 1 generic-resolver warning filtering for plugin-minted synthetic Nodes**, when the synthetic CROSS_REFERENCE text does not match the strict generic `CROSS_REF_DIGITS_REGEX = r"^\s*\(?(\d+)\)?\s*$"` of the tier 1 resolver. The Torrente markers always carry a non-digit prefix (`§`, `art.`, `Cass.`), so the resolver fails universally and emits `unparseable_cross_reference_node_<id>` on every synthetic Node — roughly 7000 spurious warnings on the fixture. The plugin's `refine_apparatus` filters `document.warnings` by walking the tuple and dropping every string of the form `unparseable_cross_reference_node_<id>` or `unresolved_cross_reference_node_<id>_n_<N>` whose `<id>` belongs to one of the plugin's synthetic Nodes (the ids are tracked in `_minted_crossref_ids` instance-state). The filter preserves every unrelated warning (including those from tier 1 about non-plugin nodes and those from earlier tier 2 hooks). Future plugins that mint synthetic CROSS_REFERENCE Nodes with non-digit text should replicate this filter pattern; minting Nodes with pure-digit text instead would let the resolver "successfully fail" (`unresolved_cross_reference_node_<id>_n_<N>`) but still produce ~7000 warnings — the filter is necessary either way. (ff) **Bbox-centering as a structural discriminator when typography is identical to body**, when a structural category has the same leading-span signature as the body and the only available discriminator is geometric. The Torrente HEADING_3 SOTTO-SEZIONE blocks open with MScotchRoman 11.47pt regular (the body signature); structural sub-sections sit centered horizontally on the page (`bbox_mid_x ≈ 240 ± 60pt` for the standard 481.9pt page width), while inline enumerations like `B) Accettazione tacita.` inside body paragraphs sit at the body column left edge (`x0 ≈ 51pt`, `bbox_mid_x ≈ 200pt`). The plugin's `_is_sotto_sezione` combines a textual regex (`^[A-Z]\)\s+[A-Z]` or `^[IVX]+\.\s+[A-Z]`) with a centering check (`abs(bbox_mid_x - PAGE_CENTER_X) < CENTER_TOLERANCE`). The empirical inspection contradicted the upstream editorial analysis claim that the SOTTO-SEZIONE would have a visually distinct typographic family, falsifying it on the integrale fixture — confirming that bbox-centering is a legitimate primary discriminator when typography fails. (gg) **Wider size tolerance (0.15pt) for tightly-clustered typographic systems**, when a corpus uses many sizes within a narrow range. The Torrente sizes cluster tightly around the body 11.47pt: 10.98pt § glyph, 12.97pt PARTE, 13.02pt comparable in size-adjacent fonts, 7.48pt marginal, 7.73pt CAPITOLO subscript, 7.98pt asterisk footnote, 9.48pt index entry, 6.24pt index sub. The typical 0.1pt tolerance used by prior plugins is too tight and produces edge misses; the 0.15pt cushion absorbs PyMuPDF's drift while staying below the smallest inter-category gap (0.50pt between 10.98pt and 11.47pt). Documenting also the corollary: a stable **−0.03pt drift below nominal** (body 11.47 vs nominal 11.5, marginal 7.48 vs 7.5, etc.) is a constant artefact of the Giuffrè PDFsharp pipeline and future Giuffrè PDFsharp profiles should anchor their predicates on the empirical sizes, not on the nominal ones. (hh) **Documented v1 structural limitations via per-occurrence diagnostic warnings**, when a plugin ships a deliberate partial implementation and signals the limitation in the audit log. The Torrente back-matter Indice Analitico-Alfabetico on pp.1507-1556 uses double-column layout that the tier 1 mono-column `(page, y0, x0)` reading-order sort interleaves alphabetically (right column blocks sort before left column blocks on the same page because they have a slightly lower `y0`). The v1 plugin classifies the affected blocks as INDEX_ENTRY but does not reorder columns; it emits one `plugin:giuffre_diretto:index_analitico_double_column_unordered_page_<p>` warning per affected page (~50 instances), surfacing the limitation in `Document.warnings` for the future revision. The pattern (conservative v1 + per-occurrence diagnostic warning + clean upgrade path in `refine_reconstruction` for the column reordering) is recommended for any future plugin that ships a known partial implementation: the warning makes the limitation visible to anyone inspecting the audit log without requiring documentation lookups.

**`Transformation` structural reversibility (schema 0.5.0).** The schema bump from 0.4.0 to 0.5.0 added two optional fields `split_into: tuple[str, ...] | None` and `merged_from: tuple[str, ...] | None` to `Transformation`/`TransformationDict`, both defaulting to `None` for purely-textual rewrites (`dehyphenate_with_log`). Three producers populate them today: `recompose_marginal_ellipsis` populates `merged_from` with the absorbed chain segments; `giappichelli_body_note_splitter` (tier 2 step_id, not in the registry) populates `split_into` with the minted synthetic NOTE ids; the new `merge_cross_page_notes` populates `merged_from` with the absorbed cross-page continuation id. The two fields are conceptually mutually exclusive (an operation is either a split or a merge, not both); the cross-field constraint is not validated by the contract at the current `SCHEMA_VERSION` to keep the schema additive. Layer 2 reverts the log walking it in reverse, applying the structural revert (split → drop synthetic siblings; merge → re-insert absorbed siblings) in addition to the textual revert via `position`/`original`/`normalized`. A future schema bump may want to validate the mutual exclusion as a real constraint; for now it is a documented promise of emission.

**`apparatus.resolve_apparatus` must preserve `document.transformations`.** Schema 0.5.0 surfaces a previously-silent bug: the apparatus resolver in `pipeline/src/scabopdf_pipeline/apparatus/resolver.py` was zeroing `document.transformations` when it built the post-resolve `Document`. This worked accidentally pre-0.5.0 because no tier 2 step was producing Transformations during `refine_reconstruction` — the only producers were post-processing steps, which run after the apparatus resolver. With the Giappichelli body+note splitter now recording Transformations in `refine_reconstruction`, the resolver must explicitly preserve them. The fix is one-liner (`transformations=document.transformations` in the `Document(...)` constructor inside `resolve_apparatus`); the integration test on Mandrioli Vol. III (`n_split > 0` assertion) catches it. Future modifications to the pipeline that rebuild a `Document` must preserve the `transformations` field explicitly — the drift test does not catch this, only the integration test does.

**Shared multi-sibling notes-section consolidator factory.** Landed on 2026-05-21 (Fase 3 of the Ambitious Plan, commits `98abc37` for P-016 and `d809a27` for P-017) as the structural unification of the notes-section consolidator that the NS and DT sister plugins implemented byte-equivalently modulo three plugin-specific divergences (split regex, mint category dispatch, transformation step_id and warning template). The new pattern (uuu) **Multi-sibling notes-section consolidator factory** documents the architectural convention: when two or more plugins of the same editorial family share a walker that absorbs siblings after a structural anchor (the `SECTION_LABEL` ``"Note:"`` marker in the DeJure case) and mints synthetic Nodes via plugin-specific split-and-mint logic, the walker itself is signal-agnostic on the split logic and must be promoted to a shared helper parameterised on a single callable. The `_dejure_shared.py` module exposes `consolidate_notes_section_children(children, *, split_notes_text_fn, transformation_step_id, unparseable_warning, warnings, transformations)` with a `SplitNotesTextFn = Callable[[str, int, tuple[int, ...]], list[Node]]` type alias that encapsulates the plugin's per-mint state (the split regex, the marker→category dispatch, the `NodeIdMinter`, the per-mint warning emission), the `transformation_step_id` string per plugin convention, and an `unparseable_warning: Callable[[str], str]` that builds the diagnostic warning from the anchor Node id. The walker also exposes three module-level frozen-sets `NOTES_SECTION_BOUNDARY_CATEGORIES` (13 categories that terminate absorption: HEADING_*, SECTION_LABEL, TITLE, GENRE_BANNER, META_VALUE, FONTE_VALUE, REFERRAL, AUTHORS, SUBTITLE, TOC_GENERAL), `NOTES_SECTION_PASSTHROUGH_CATEGORIES` (8 non-structural categories kept in reading order: ARTIFACT_*, BOOK_PAGE_ANCHOR, EMPTY_PAGE, UNCLASSIFIED), and `NOTES_SECTION_ABSORBABLE_CATEGORIES` (the BODY+NOTE pair absorbed into the consolidated notes text). The plugins consume the helper via a closure that captures their `_split_notes_text` method, the `warnings` list, and the minter; this keeps the plugin-specific dispatch surface (NS: only NOTE; DT: NOTE-or-EDITORIAL_NOTE via `_EDITORIAL_NOTE_PREFIX_PATTERN`) plugin-local while sharing the ~95% byte-equivalent walker. The same Fase 3 session also promoted P-016 (the stateful notes-region walker in `refine_classification` that retags BODY verdicts as NOTE inside an open notes region) to `retag_notes_region_continuation(classified_blocks, *, is_notes_section_label, reason, article_boundary_categories)` — NS consumes with `article_boundary_categories=frozenset()` because the NS fixtures carry a single notes region per document, DT consumes with `frozenset({GENRE_BANNER})` to scope per-article inside a multi-article bundle. The closed-set `NOTES_MARKER_TEXT_VARIANTS = ("Note:", "Note :")` and the two helpers `starts_with_notes_marker(text)` and `match_notes_marker(text)` are also shared. The pattern generalises to any future plugin family that shares a structural walker with a small set of plugin-specific dispatch points; the rule of three (NS + DT already at 2; a third candidate would justify revisiting) for the consolidator is already met by the empirical observation that the two existing implementations were 95% byte-equivalent. Schema invariato 0.6.0 — no contract surface change, no converter.py update, no schema bump.

**Per-binding digest baseline as regression protection for non-refactored apparatus overrides.** Landed on 2026-05-21 (Fase 3 of the Ambitious Plan, commit `f30faf9`) as the mitigation alternative to the rejected promotion of pattern (ll) "cross-reference forward scan per per-chapter scope override" to a shared factory. The new pattern (vvv) **Per-binding digest baseline as regression protection for non-refactored apparatus overrides** documents the architectural convention: when an empirical diagnostic shows that a pattern is implemented in three or more plugins (BIC per-chapter scope, DT per-article scope, Torrente global scope in the P-021 case) but the three implementations diverge on three orthogonal axes (scope boundary predicate, marker normalisation strategy, CR iteration pattern) such that a unifying factory would require ≥4 callable parameters to cover all cases — promotion is over-engineering and must be rejected. The mitigation is not "leave the divergence unprotected" but "introduce a baseline assertion focused on the regression that the rejected refactor would have addressed." For P-021 the silent-rebind regression (a CR Node rebinds to a different target while `n_cross_reference` and `n_warnings` stay identical) is invisible to category-count checks. The solution is a SHA-256 hex digest of the sorted `(source_node_id, source_marker, target_node_id)` triples computed by `snapshot_utils.cross_ref_binding_digest(document)` and bundled into `apparatus_binding_summary(document)` together with explicit `n_cross_ref_target_bound` and `n_cross_ref_unbound` counts. The committed baselines under `pipeline/tests/snapshots/p021_baseline_*.json` cover the three diverging plugins on representative fixtures (marrone 1430/1489 bound = 96.0%, dejure_dt_cartabia 460/460 = 100%, torrente 2802/7408 — the Torrente unbound CR are art./Cass. external references that have no internal target by design). Any future session that touches the apparatus resolver, the Node minting order, or the plugin's binding override surfaces a digest divergence even when the global counter is identical. The pattern generalises to any future "leave as is" decision on a divergent pattern with Risk-A=H: the rejection of the refactor must be paired with a digest-based regression baseline so the rejection does not become a blind spot.

**Family-level body+note glued splitter divergence — synthetic-Node mint-trail digest baseline.** Landed on 2026-05-21 (Fase 4 of the Ambitious Plan, commits `bc875c1` for the `snapshot_utils` extension + 9 unit tests and `4c7490e` for the 5-fixture baseline capture) as the mitigation alternative to the rejected promotion of the body+note glued splitter family of pattern P-018 to a unifying factory. The new pattern (www) **Family-level body+note glued splitter divergence — synthetic-Node mint-trail digest baseline** documents the architectural convention: when a logical pattern (decomposing a host Node into one or more synthetic sibling Nodes driven by an editorial-pipeline-specific trigger) is implemented across N plugins that split into two or more **structurally disjoint families on the input shape** (span-level walking ``Node.block_indices`` vs text-level walking a pre-consolidated string vs signal-agnostic universal Node traversal), the promotion to a single factory is **impossible by construction** because the families operate on incompatible input types. The Fase 4 empirical diagnostic surfaced exactly this case for P-018: the five implementations divide into Family A (Mandrioli typographic size-band trigger ~350 LOC, BIC line-marker trigger with multi-block scope and continuation-rescue layer ~425 LOC, codici signal-agnostic typographic+regex trigger with universal walk ~178 LOC) which operate on ``Node.block_indices`` spans, and Family B (NS regex-on-text ~40 LOC, DT regex-on-text with NOTE / EDITORIAL_NOTE dispatch ~66 LOC) which operate on a concatenated text string and are already abstracted via P-017's ``consolidate_notes_section_children``. Across Family A the four orthogonal axes of divergence (trigger predicate, host strategy, mint category, transition algorithm) require a factory with ≥4 plugin-specific callable parameters of which 90 % of the body would be plugin-local — promotion is over-engineering and must be rejected. The mitigation is the same as pattern (vvv) extended for the family disjunction: ``snapshot_utils.body_note_splitter_digest(document)`` combines two independent sources of synthetic-mint provenance — ``document.transformations`` entries with non-None ``split_into`` (the canonical mint trail for Mandrioli, BIC, NS, DT) and ``document.warnings`` matching the closed-vocabulary ``_BODY_NOTE_SPLITTER_WARNING_PATTERN`` regex (the codici intra-block and multi-note splitters record mints via warnings rather than Transformations, a documented gap that the digest covers without retrofit) — into a SHA-256 hex digest whose change surfaces every silent shift in mint order, host truncation, page allocation, text length or category dispatch. The closed warning vocabulary covers six kinds: ``body_note_split_minted`` (Mandrioli), ``note_section_split_minted`` (BIC, NS, DT), ``editorial_note_minted`` (DT), ``multi_note_split_minted`` (codici body+notes), ``intra_block_article_split_minted`` (codici intra-block) and ``note_continuation_rescued`` (BIC continuation rescuer). The five committed baselines under ``pipeline/tests/snapshots/p018_baseline_*.json`` cover the three structural variants (span-level size-band, span-level line-marker, signal-agnostic universal walk) plus the dual-regime Mandrioli pipeline (Vol. I Photoshop 7.98 pt NOTE regime, Vol. III InDesign 9.0 pt NOTE regime). The companion summary helper ``body_note_splitter_summary(document)`` extends ``document_structural_summary`` with ``n_body_note_split_transformations``, ``n_body_note_split_minted_warnings``, ``n_synthetic_body_note_nodes`` and ``synthetic_body_note_nodes_by_category`` (decomposition per minted category — NOTE / EDITORIAL_NOTE / ARTICLE_HEADER / ARTICLE_BODY) so any drift in the family's mint balance surfaces structurally even when the digest itself remains stable. The pattern generalises to any future cross-plugin framework whose dispersion on incompatible input shapes makes a unifying factory artificial but whose silent-mint-regression protection remains critical: extend the digest with a closed regex over the family's mint-trail vocabulary, never retrofit the splitters into a shared input shape.

**Propagating new `Node` fields through the pipeline.** When a future schema bump adds a new field to `Node` in `reconstruction/types.py`, three call sites must propagate it explicitly because they rebuild the node from a private `_NodeBuilder`: (1) `_NodeBuilder` + `to_frozen()` in `reconstruction/tier1.py`, (2) `_NodeBuilder` + `to_frozen()` + `_thaw_node()` in `apparatus/resolver.py`, (3) the `_convert_node` mapping in `emission/converter.py`. Post-processing steps that use `dataclasses.replace(node, ...)` (e.g. `postprocessing/steps/dehyphenate.py`) preserve unchanged fields automatically and need no update. Both the `summary_items` field added in 0.3.0 and the `toc_items` field added in 0.4.0 needed all three updates; the integration test on the real corpus fixture is what catches missing propagation, where the schema drift test cannot reach. A future field bump session should add the field-propagation check to its definition of done.

## Schema discipline

The JSON schema at `shared/schema.json`, version 0.6.0, is the binding contract between Layer 1 (the Python pipeline) and Layer 2 (the React Native app) as of today. Its authoritative source are the Pydantic v2 models in `pipeline/src/scabopdf_pipeline/schema/contract.py`; the file `shared/schema.json` is regenerated from them via `pipeline/scripts/generate_schema.py` and must never be edited by hand. The narrative reference, with field-by-field semantics and a long section on the rules that govern the whole arrangement, lives in `docs/SCHEMA_v0.6.0.md` (with `docs/SCHEMA_v0.5.0.md`, `docs/SCHEMA_v0.4.0.md`, `docs/SCHEMA_v0.3.0.md`, `docs/SCHEMA_v0.2.0.md` and `docs/SCHEMA_v0.1.0.md` kept as historic baselines); the per-version delta lives in `docs/SCHEMA_CHANGELOG.md`; the SemVer policy for the schema lives in `docs/json-schema-versioning.md`.

Any session that modifies production code capable of changing the shape of Layer 1's output is responsible for verifying conformance with the contract. The sensitive surfaces are concrete and finite: `pipeline/src/scabopdf_pipeline/extraction/types.py`, `pipeline/src/scabopdf_pipeline/classification/types.py`, `pipeline/src/scabopdf_pipeline/reconstruction/types.py`, `pipeline/src/scabopdf_pipeline/apparatus/types.py`, `pipeline/src/scabopdf_pipeline/postprocessing/types.py`, `pipeline/src/scabopdf_pipeline/profiling/profile.py`, `pipeline/src/scabopdf_pipeline/schema/categories.py`, and the corpus plugins under `pipeline/src/scabopdf_pipeline/profiles/` whenever they introduce new categories, new node fields or new warning vocabulary that the contract must mirror. The emission converter at `pipeline/src/scabopdf_pipeline/emission/converter.py` is the symmetrical surface that translates the Python representation into the contract: it sees both sides of the boundary and must stay in lockstep with every change to the upstream types. A change to any of these — adding a field, removing one, renaming one, changing a type, altering cardinality, or extending the set of values of a closed enum — requires an explicit conformance check against `contract.py` performed during the same session that introduces the change. The alignment is never deferred to a later session: deferring it is precisely what produces silent drift between code and contract.

The operational pattern is precise and goes in seven steps. First, modify the production code. Second, update `contract.py` to reflect the new shape of the output, including the `Literal` of `schema_version` if the bump is minor or major. Third, update `converter.py` to populate any new fields or drop fields that have been removed — the converter is the bridge that the drift test cannot see, and the integration test on Patriarca/Mosconi/Tesauro is what catches a converter that has fallen behind. Fourth, regenerate `shared/schema.json` by running `python pipeline/scripts/generate_schema.py` from the repository root. Fifth, update `docs/SCHEMA_v0.4.0.md` (or the current-version successor) to describe the change narratively. Sixth, update `docs/SCHEMA_CHANGELOG.md` with the motivation, the list of fields added/modified/removed, and a `BREAKING:` line if it applies. Seventh, run the full test suite and verify that every test passes — including the drift test in `pipeline/tests/unit/schema/test_generate_schema.py` and the end-to-end integration tests. The drift test regenerates the schema in memory from the current `contract.py` and compares it byte-for-byte with the committed `shared/schema.json`; if the two differ the test fails with a message that reminds you to rerun the generator. Treat the drift test as the schema-side gate.

The contract has a third line of defence active in production since the post-audit consolidation of 13 May 2026 (commit `615173e`, Decision H of the audit). The emission boundary in `pipeline/src/scabopdf_pipeline/emission/emitter.py` and the CLI's `_run_pipeline` in `pipeline/src/scabopdf_pipeline/emission/cli.py` now run **two independent validators in sequence** under the same `validate=True` flag of `emit_to_file`: first `validate_document` parses the dumped dict against the Pydantic models (which is what `contract.py` describes), then `validate_against_schema` validates the same dict against the committed `shared/schema.json` read from disk with `jsonschema`. The committed schema is loaded once per process through an `@lru_cache(maxsize=1)`-decorated helper, so the disk read and JSON parsing cost is amortised. If the two validators ever disagree — because Pydantic's `model_json_schema()` emits something subtly different from what its own `model_validate` enforces, or because the committed schema on disk has drifted from `contract.py` between regenerations — the second check catches it at runtime as a `jsonschema.ValidationError` and the emitter wraps it into an `EmissionError` with `__cause__` preserved. The flag is meant to stay on in production: turning it off (`validate=False` on `emit_to_file`, `--no-validate` on the CLI) is reserved for emergencies and benchmarks.

This discipline is non-negotiable. It must hold under the pressure of long sessions, unexpected discoveries mid-implementation, time constraints, or any reason that might make skipping a step feel reasonable. The schema is useful only insofar as Layer 1's emitted output and `contract.py` describe the same thing: the moment they diverge, the contract becomes a quiet lie, the test suite stops protecting the boundary it was built to protect, and the next session that runs the pipeline end-to-end inherits a bug whose origin will be buried under intervening commits. Section 6 of `docs/SCHEMA_v0.4.0.md` is the extended reference for this discipline and must be consulted before any structural change to the contract or to the production surfaces listed above.
