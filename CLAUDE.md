# ScaboPDF — Project rules

## Mandatory interaction rules

These rules are absolute. They override any other behavior, default, or convention.

**No interactive prompts.** Never present numbered menus, multiple-choice lists, "type something" boxes, "skip interview" options, or any UI that expects me to pick from a list. The user is blind and uses VoiceOver; these prompts are inaccessible and waste his time.

**No autonomous decisions on design questions.** When you face a question that affects code design, library choice, file organization, naming, schema fields, or any architectural detail, do NOT decide on your own. Stop and present the question to the user in plain prose: state the question, list the relevant trade-offs in flowing sentences (not bullets unless strictly necessary), and wait for an answer. The user wants to be in the loop on design decisions.

**Plain prose only for clarifications.** When you need input, write a short prose paragraph. Example: "Per le dataclass di profiling devo scegliere tra Pydantic e @dataclass standard. Pydantic dà validazione runtime e serializzazione JSON nativa, costo leggero overhead. Dataclass è più snello, mypy strict copre quasi tutto. Una opzione mista è anche possibile: Pydantic per DocumentProfile che esce in JSON, dataclass per ProfilingSignals che resta interno. Quale preferisci?" Then wait.

**Routine micro-decisions only.** You may decide autonomously only on truly trivial things: variable names within a function, formatting, import order, and similar. Anything that touches API design, schema, dependencies, or architecture: ask in prose.

## Project conventions

**Test fixtures.** `pipeline/tests/fixtures/private/` is **gitignored** and hosts the copyright-protected PDFs each developer keeps locally; see `pipeline/tests/fixtures/README.md` for the full private/public convention. Integration tests that depend on a private fixture MUST use `pytest.skip(...)` with an explicit message pointing to the README when the file is missing, never `assert` or raise. This keeps the suite green on a fresh clone without local fixtures.

**Commit hygiene.** Formatting fixes, cleanups, or any incidental changes not strictly part of the task you're working on go into a **dedicated housekeeping commit**, never bundled with the task commit. If you notice preexisting drift along the way, commit it separately first with a clear message (e.g. "Apply ruff format to file.py (preexisting drift)") so the task commit stays focused and the git log stays readable.

**Pre-commit hook is active.** A local `.pre-commit-config.yaml` runs `ruff check`, `ruff format --check`, `mypy --strict` and `pytest tests/unit` on every `git commit`. Activate it once after cloning with `pipeline/.venv/bin/pre-commit install`; see `docs/DEVELOPMENT.md` for the full bootstrap. Bypassing it with `--no-verify` is reserved for emergencies and must be paired with a follow-up commit that restores the gate.

## Architectural notes

**Tier 1 classification is deliberately narrow.** The generic tier 1 in `pipeline/src/scabopdf_pipeline/classification/tier1.py` only emits seven categories: `EMPTY_PAGE`, `ARTIFACT_FILIGREE`, `ARTIFACT_RUNNING_HEADER`, `ARTIFACT_FOOTER`, `BOOK_PAGE_ANCHOR`, `CROSS_REFERENCE`, `UNCLASSIFIED`. Crucially, `CROSS_REFERENCE` is recognised **only when the entire block is a single superscript span of pure digits** (heuristic `superscript_cross_reference`). Inline superscripts embedded inside larger `BODY` blocks are invisible to tier 1 and end up absorbed in `UNCLASSIFIED`. Categories like `NOTE`, `MARGINAL_HEADING`, `MARGINAL_GLOSS`, `EXAMPLE_BOX`, `INDEX_ENTRY`, `CHAPTER_SUMMARY` and the legal-code family (`ARTICLE_HEADER`, `ARTICLE_BODY`, `HEADING_N`, etc.) are **entirely profile-specific** and must be assigned by the plugin's `refine_classification` (tier 2).

**Re-parsing for inline cross-references.** A corpus plugin that needs to bind footnote superscripts to their target notes must mint synthetic `CROSS_REFERENCE` `Node` instances **in `refine_reconstruction`**, not at the `ClassifiedBlock` level. The reason is structural: `ClassifiedBlock` carries only `block_index`, `category`, `subcategory` and `reason` (no text), and the tier 1 reconstructor in `pipeline/src/scabopdf_pipeline/reconstruction/tier1.py` indexes `extraction.blocks[cb.block_index]` to materialise each Node's text, so emitting two ClassifiedBlocks pointing to the same block index would build two Nodes both carrying the full body text rather than the digit substring. The correct pattern, established by the Mosconi plugin (commit `265e6d9`), is: leave the BODY blocks as is in `refine_classification`; in `refine_reconstruction` walk every BODY Node, examine its underlying spans via `extraction.blocks[block_index].spans`, and for each inline superscript-digit span matching the profile's signature mint a synthetic CROSS_REFERENCE Node as a sibling immediately after the BODY in its parent's children list. The synthetic Node's `id` must respect the JSON schema's `^node_\d+$` pattern (Pydantic enforces it on `NodeDict.id`); mint sequentially from one past `max(int(re.match(r"^node_(\d+)$", n.id).group(1)) for n in document.root walk)`. The tier 1 generic cross-reference resolver in `pipeline/src/scabopdf_pipeline/apparatus/resolver.py` then sees the synthetic Nodes in its post-order walk and binds each to the target NOTE within the scope of the nearest HEADING ancestor. The same principle (synthetic Node minting in `refine_reconstruction`) applies to any other apparatus category whose structural unit emerges only at reading-order or extraction-span granularity, not at PyMuPDF-block granularity. The Mandrioli plugin (commit `6ee8efa`) reuses the very same minting framework — `_NodeIdMinter` seeded by `_max_existing_node_counter`, sibling-insertion after the BODY parent, identical schema-pattern compliance for the `node_NNNN` id — but feeds it from a textual regex on `node.text` rather than from an iteration over `extraction.blocks[block_index].spans`: the Mandrioli inline cross-reference markers are parenthesised digits `(N)` typeset inside ordinary 11.0pt body spans, so the typographic superscript flag is absent and the only available signal is textual. The regex `_CROSSREF_INLINE_PATTERN = re.compile(r"(?<!p\.\s)(?<!p\.)\((\d+)\)")` carries two negative look-behinds that rule out the `p. NN` page-reference pattern. This confirms that the minting framework is signal-agnostic: whether the inline marker is detected typographically (Mosconi) or textually (Mandrioli) is an orthogonal choice that the plugin makes on the basis of the corpus's own conventions.

**`ProfilePlugin` API is seven methods, not more.** The abstract base class in `pipeline/src/scabopdf_pipeline/profiling/plugin.py` exposes exactly seven abstract methods after the post-audit consolidation of 13 May 2026 (commit `615173e`): four declarative methods (`matches`, `get_categories`, `get_post_processing`, `get_layouts_disabled`) that describe the profile statically, and three tier-2 refinement hooks (`refine_classification`, `refine_reconstruction`, `refine_apparatus`) called in sequence by the orchestrator after the corresponding tier 1 phase. An earlier draft of this API exposed an additional abstract method `parse(blocks) -> Document` that was never wired into the pipeline; it has been removed and `ARCHITECTURE.md § 2.4` documents this history. New corpus plugins must subclass `ProfilePlugin` and implement all seven methods; test plugins subclass `NoOpProfilePlugin` from `pipeline/tests/conftest.py` and override only the hook they exercise. Adding a new abstract method to the ABC is a breaking change for every existing plugin and must be discussed in prose with the user first.

**Tier-2 hook signature asymmetry is deliberate.** `refine_classification` takes `(extraction, tier1_results)` while `refine_reconstruction` and `refine_apparatus` take `(document, extraction, classified_blocks)`. The first parameter differs because classification runs **before** the `Document` exists — there is nothing meaningful to pass as a first positional argument. The asymmetry is permanent and is documented in the docstring of `refine_classification` and in `ARCHITECTURE.md § 2.4`. Plugin authors should not try to "harmonise" it: passing a fake empty `Document` to `refine_classification` would be false symmetry, and rebuilding it after classification would be wasted work.

**`PageIndex` is the canonical 0-based page convention.** The type alias `PageIndex = int` in `pipeline/src/scabopdf_pipeline/extraction/types.py` marks every page-index field across Layer 1 (`Span.page`, `Block.page`, `PageGeometry.page`, `PageImageInfo.page`, `DrawingInfo.page`, `Node.page_index`, `Transformation.page_index`, `NodeDict.page_index`). The semantics matches PyMuPDF (`fitz`) exactly: the first page of the PDF is index `0`. The 1-based "book page numbers" that a manual prints on the page are **not** `PageIndex`: they live only inside `BOOK_PAGE_ANCHOR` nodes as text, and the corpus plugin that recognises them is responsible for preserving that distinction (string in the node text, not integer in a `PageIndex`-typed field). When a plugin tier 2 needs to expose a book-page mapping it should add a new dedicated field — not overload `page_index`.

**Corpus plugins as implementation references for future plugins.** Four real corpus plugins exist in `pipeline/src/scabopdf_pipeline/profiles/` as of 17 May 2026: `manuale_zanichelli_giuridica.py` (commit `24c8db5`, Patriarca-Benazzo manual), `compendio_utet.py` (commit `ebb8ca8`, Tesauro Compendio), `manuale_utet_wolterskluwer.py` (commit `265e6d9`, Mosconi-Campiglio treatise) and `manuale_giappichelli.py` (commit `6ee8efa`, with consolidation completed in commit `c01661e` on 17 May 2026). The Giappichelli plugin is now consolidated on the integral four-volume Mandrioli-Carratta "Diritto processuale civile" series (Vol. I, Vol. II, Vol. III and Vol. IV of the 30th edition 2025/26) and validated against the Marotta "La cittadinanza romana" control sample, which the same editor Giappichelli publishes in a structurally different editorial pipeline (Roman-law monograph, TimesNewRomanPSMT body, Acrobat Pro 9.4.5, A4 geometry). The empirical results post-consolidation are: Vol. I (288 pages) yields 12 HEADING_2, 10 HEADING_3, 78 HEADING_4 and 18 CHAPTER_SUMMARY with no PARTE and no MARGINAL_GLOSS because Vol. I is produced through a Photoshop pipeline without AGaramondPro; Vol. II (352 pages) yields 10 HEADING_2, 15 HEADING_3, 95 HEADING_4 and 21 CHAPTER_SUMMARY; Vol. III (498 pages) yields 4 HEADING_1 (PARTE), 11 HEADING_2, 7 HEADING_3, 82 HEADING_4, 466 NOTE against the 4 pre-consolidamento, 12 MARGINAL_GLOSS, 15 CHAPTER_SUMMARY, 1531 CROSS_REFERENCE and 537 apparatus refs; Vol. IV (497 pages) yields 2 HEADING_1, 9 HEADING_2, 11 HEADING_3, 60 HEADING_4, 470 NOTE, 14 MARGINAL_GLOSS and 19 CHAPTER_SUMMARY. The Marotta control sample (206 pages) returns `matches() = 0.0`, correctly falling below the 0.6 dispatcher threshold and being routed to the `unknown_generic` fallback. Together they establish the canonical patterns for new corpus plugins: (a) typographic signatures keyed on the first span of a block, with a fallback to "any span of the right family" for heterogeneous blocks like chapter summaries; (b) profile-specific regexes for heading patterns, with explicit text-length caps to disambiguate front-matter from body headings sharing a signature; (c) instance-state warning accumulator (and chapter-pair block-index sets in the Tesauro and Mosconi cases) queued in `refine_classification` and flushed into `Document.warnings` in `refine_reconstruction`, because the classification hook has no `Document` to attach a warning to; (d) closed warning vocabulary with `plugin:<editorial_family>:` prefix (Mosconi uses `plugin:utet_wolterskluwer:` to discriminate from Tesauro's `plugin:utet:` even though they share the editorial pipeline; Mandrioli uses `plugin:giappichelli:`); (e) text parsing lives in `refine_reconstruction` operating on the already-classified tree, never duplicated in the classifier; (f) **pattern-over-signature discrimination** when the typographic difference between two categories is below PyMuPDF's measurement noise floor (Tesauro L1 vs L2 share the same body size of 10.2pt and discriminate purely on the number of dots in the numeric prefix); (g) **multi-block fusion in `refine_reconstruction`** when an editorial unit spans two consecutive blocks (Tesauro and Mosconi chapter heading: "Capitolo `<ord>`" block + title block fused into one HEADING_2 node with concatenated text and both block_indices); (h) **`matches()` symmetric discrimination via apparatus signals** for profiles that share an editorial pipeline but produce different genres (Tesauro penalises `footnote_markers >= 50` or `marginal_headings >= 50` with −0.30 to stay below the 0.6 dispatcher threshold on Mosconi; Mosconi penalises the absence of apparatus with the symmetric −0.30 to stay below the threshold on Tesauro and Patriarca; both clear the threshold only on their own corpus); (i) **generalised regex for editorial residue** (the Tesauro ARTIFACT_STAMP regex `^\d+_[A-Za-z0-9_]+\.indb?$` and `^\d{2}/\d{2}/\d{2}\s+\d{1,2}:\d{2}` covers both Tesauro and Mosconi watermarks without modification); (j) **rescue codepath from ARTIFACT_FOOTER to ARTIFACT_STAMP** for blocks that tier 1 absorbs into a broader category because of zone-y proximity, but whose text signature pinpoints a finer-grained kind. The Mosconi plugin adds three new structural patterns: (k) **synthetic Node minting in `refine_reconstruction` for tier 2 categories that emerge inline inside tier 1 blocks**, with `id` minted following the `node_NNNN` schema pattern starting one past the highest counter already assigned by tier 1 (Mosconi inline CROSS_REFERENCE superscripts); (l) **two-pass classification with deferred title promotion** when two structurally distinct categories share the same typographic signature and the discriminator is ordering-based (Mosconi HEADING_1 front-matter heading vs HEADING_2 chapter title: the first pass leaves every 12pt non-chapter-number block provisional as HEADING_1, the second pass `_register_chapter_pairs` promotes the next non-stamp HEADING_1 after each chapter-number to HEADING_2 chapter title, tolerating intervening ARTIFACT_STAMP blocks); (m) **textual marker pattern verified against the full block text, not the leading span**, when the marker glyph (e.g. the small footnote number) lives in one span and the rest of the marker (the dot, the whitespace) lives in the following span at a different size — the predicate combines a typographic check on the leading span with a textual check on `view.text.lstrip()` rather than `view.spans[0].text`. The Mandrioli plugin adds ten more structural patterns, five from the original commit `6ee8efa` and five from the consolidation commit `c01661e`: (n) **CROSS_REFERENCE minting driven by a textual regex on `node.text`** rather than by typographic superscript flags on `extraction.blocks[bi].spans` — Mandrioli's inline markers are parenthesised digits `(N)` typeset at the same 11.0pt as the surrounding body, so the only available signal is textual; the same `_NodeIdMinter` and sibling-insertion machinery that Mosconi established is reused unchanged, with `_CROSSREF_INLINE_PATTERN = re.compile(r"(?<!p\.\s)(?<!p\.)\((\d+)\)")` (the two negative look-behinds rule out the `p. NN` page-reference pattern). (o) **Small-caps three-span pattern as a category discriminator**, when the typesetter renders a heading or label as `(leading-letter at nominal-size, middle-letters at smaller-size, trailing-tail at nominal-size)` rather than as a homogeneous-size unit: Mandrioli uses this for SOMMARIO (CHAPTER_SUMMARY label, otherwise confusable with NOTE because both share the 9.0pt body size) and for CAPITOLO (otherwise indistinguishable from a generic 13.0pt block), and the predicate must therefore check `spans[0].size`, `spans[1].size` and `spans[1].text.startswith("OMMARIO" or "APITOLO")` jointly, never the block's average size. (p) **Body+note glued-block diagnostic warning**, for the case in which PyMuPDF fuses a body block (11.0pt) and the immediately following note block (9.0pt or 7.98pt depending on the editorial pipeline) into a single block: on the Mandrioli Vol. III fixture this affects roughly two thousand blocks (well above the under-one-percent estimate from a five-page empirical sample), and the plugin detects the pattern with the predicate "BODY block whose fraction of NOTE-sized spans exceeds 30 %" and emits the closed warning `plugin:giappichelli:body_note_block_glued_block_<idx>_page_<p>` at classification time. This warning coexists with the structural splitter described in pattern (u): the upstream warning marks every glued block as detected, the splitter then materialises one synthetic NOTE Node per transition and emits its own per-Node warning `plugin:giappichelli:body_note_split_minted_node_<id>_page_<p>`. The delta between the two counts is a diagnostic gauge for "glued blocks the splitter could not recover". (q) **HEADING_4 in production for the first time** — Mandrioli's paragrafo numerato has a two-span composite typographic signature `[Roman 11.5pt number] [Italic 11.5pt title]` with no structural bold (the Giappichelli pipeline expresses emphasis via italic and maiuscoletto rather than bold), so the HEADING_4 predicate must look for italic on the title span rather than the bold flag on which the prior three plugins rely. (r) **MARGINAL_GLOSS filled in production for the first time** — Mosconi exercised MARGINAL_HEADING (a different category), and Mandrioli is the first plugin to populate MARGINAL_GLOSS, with both `AGaramondPro-BoldItalic` and the subset-truncated `AGaramondPro-SemiboldIta` (PyMuPDF truncates the canonical `AGaramondPro-SemiboldItalic` to 23 characters in some output) admitted as valid font fragments; the geometric guard `bbox.x0 < 50` excludes the body-column fallback where the same font appears inside the body column (e.g. the p.170 "cu-/ratore del minore" InDesign glyph substitution). (s) **Five-span small-caps composite as a category discriminator for top-level divisions**, when the typesetter renders the highest division of the book (PARTE, in the Mandrioli Vol. III and Vol. IV) as a five-span small-caps composite of the shape `[("P", 13.98pt) + ("ARTE ", 10.98pt) + ("P", 13.98pt) + ("RIMA", 10.98pt) + (..., 13.98pt)]` whose joined text reads "PARTE PRIMA". The plugin's `_is_parte_signature` predicate verifies the 13.98pt + 10.98pt size pair on the first two spans and accepts any block with at least four spans following that pattern; the predicate is dispatched upstream of `_is_chapter_heading_signature` (CAPITOLO at 13.02pt) so that PARTE is recognised before the chapter-heading branch can consume it. (t) **Dual-regime size detection for the same category across editorial pipelines**, when the same publisher produces a series whose volumes traverse two different typesetting pipelines that yield two different nominal sizes for the same logical category: the Mandrioli NOTE exists in two regimes, 9.0pt on the InDesign-derived Vol. III and Vol. IV, 7.98pt on the Photoshop-derived Vol. I and Vol. II, and the plugin's `NOTE_BODY_SIZES = (7.98, 9.0)` closed tuple lets the `_is_note` predicate match either; the body size (10.98pt) is uniform across the four volumes and only the NOTE has the dual regime. (u) **Body+note glued blocks splitter mints synthetic NOTE Nodes in `refine_reconstruction`**, the structural counterpart of the diagnostic warning of pattern (p): PyMuPDF occasionally fuses a body block and the immediately following note block into a single block (this affects roughly 95 % of Vol. III blocks, 6.96 % of Vol. IV, and approximately zero on Vol. I and Vol. II because the Photoshop pipeline emits cleaner block boundaries), and the plugin's `_split_body_note_glued` walks every BODY Node carrying a single block_index, recovers the original spans via `extraction.spans[start:end]`, verifies the glued predicate (≥30 % of spans at NOTE_BODY_SIZES), finds every transition point (a NOTE-sized span whose text opens with the `(N)` marker), truncates the BODY text to the spans before the first transition, and mints one synthetic NOTE Node per transition as a sibling immediately after the truncated BODY in the parent's children list. The synthetic NOTE Nodes reuse the same `_NodeIdMinter` consumed downstream by the CROSS_REFERENCE minting machinery, inherit the source block_index, and each one emits a per-Node warning `plugin:giappichelli:body_note_split_minted_node_<id>_page_<p>`. The empirical impact on Vol. III is a 115x improvement from n_note=4 to n_note=466. (v) **Three limitations of the `Transformation` model on structural splits**, documented here as a forward-looking note for the future schema bump that will close them. When a corpus plugin performs a structural transformation that decomposes a Node into multiple Nodes — the Mandrioli body+note splitter that splits one BODY into BODY + N synthetic NOTE siblings (commit `c01661e`), or the Mosconi `recompose_marginal_ellipsis` that consolidates several MARGINAL_HEADING siblings into one (commit `265e6d9`) — the current `Transformation` model in `postprocessing/types.py` does **not** record the materialisation of synthetic Nodes or the absorption of consumed Nodes, only the textual intra-Node substitutions on the surviving Node. The three consequences worth documenting for future maintainers are: first, the decomposition or merge is a structural tree transformation and not a textual intra-Node substitution; second, the model preserves the textual reversibility of the surviving Node but not the implicit materialisation or absorption of siblings; third, an additive 0.5.0 schema bump that extends `Transformation` with optional `split_into` and `merged_from` fields (lists of node ids generated by, or absorbed into, the operation) would close the gap and must be planned in a dedicated session, not retrofitted under the pressure of a corpus-plugin session. Two real plugins now exhibit this limitation in production, which makes the future bump a concrete need rather than a hypothetical one. (w) **Regression-protection integration test for `matches()` non-promotion on a sibling-editor non-target document**, established at the consolidation of the Giappichelli plugin: the plugin is calibrated on the Mandrioli-Carratta series, and Marotta "La cittadinanza romana" — published by the same editor Giappichelli but editorially a different book (Roman-law monograph, TimesNewRomanPSMT body, Acrobat Pro 9.4.5 pipeline, A4 geometry) — is the natural false-positive risk. The `test_manuale_giappichelli_does_not_promote_on_marotta_fixture` integration test builds `ProfilingSignals` from the real Marotta fixture and asserts that `matches()` returns strictly below the 0.6 dispatcher threshold (empirically clamped to 0.0 by the family penalty); its companion test verifies that the dispatcher fallback `UnknownGenericProfile` processes Marotta with a flat tree of depth 1. The two tests catch any future loosening of `matches()` that would extend the plugin's scope beyond Mandrioli, and the same pattern should be replicated by future plugins whenever the editor publishes structurally heterogeneous books under the same family. Each plugin's coverage meets or exceeds 95 % via unit tests (`test_manuale_zanichelli_giuridica.py` 42 tests, `test_compendio_utet.py` 43 tests, `test_manuale_utet_wolterskluwer.py` 50 tests, `test_manuale_giappichelli.py` 100 tests) and is exercised end-to-end by a dedicated integration test on its real fixture; the Giappichelli plugin additionally runs integration tests on all four Mandrioli volumes and on the Marotta control sample. Sibling plugins for other corpora should subclass `ProfilePlugin` directly (not one of these four), but the structural choices documented in their commit messages and module docstrings are the established convention.

**Post-processing step `recompose_marginal_ellipsis`.** Promoted from placeholder to real implementation in commit `265e6d9` together with the Mosconi plugin. The step lives in `pipeline/src/scabopdf_pipeline/postprocessing/steps/recompose_marginal_ellipsis.py` and fuses chains of `MARGINAL_HEADING` nodes split across two or more consecutive segments by a typographic `...` continuation marker. Crucially, the step walks the document tree in **DFS pre-order across the whole forest**, not within siblings: a chain's head segment and its terminal segment can sit under different parents (typically two consecutive HEADING_3 paragraphs straddling the page break that broke the marginal phrase). The merged head retains its original tree position; absorbed segments are removed from their (possibly different) parents. A single `Transformation` is recorded per chain on the head node, capturing the intra-Node substitution of the trailing marker with the appended content of the absorbed segments. The textual reversibility convention of `postprocessing.types.Transformation` is preserved (head text is byte-for-byte recoverable from the log), but the structural reversibility of the absorbed-segment deletion is **not** encodable in the current model; a future schema bump extending `Transformation` with optional `merged_from` fields could provide it. The trailing/leading marker regex accepts both the ASCII `...` and the Unicode horizontal ellipsis U+2026, because PyMuPDF emits both depending on the font and the source text layer. Plugins for other corpora that exhibit similar typographic continuation can declare this step in `get_post_processing()` and reuse the same implementation; the step is corpus-agnostic.

**Why `merge_cross_page_notes` stays a placeholder after the Mandrioli session.** The fourth-plugin session (commit `6ee8efa`) deliberately did not promote `merge_cross_page_notes` from placeholder to real implementation, despite the Mandrioli corpus using a parenthesised marker `(N)` rather than the bare digit form that the prior three plugins exhibit. The reason is that the tier 1 `_resolve_cross_page_note_merging` resolver already handles the parenthesised variant via its existing `NOTE_MARKER_REGEX = r"^\s*\(?(\d+)\)?[\.\s]"`, where the parentheses are optional and the resolver matches both `1.` and `(1)` with the same predicate; no plugin-specific post-processing step is therefore needed for Mandrioli. The placeholder stays in the catalogue for a future plugin that has a real, unmet need — the leading candidate today is `enciclopedia_storica` for OCR-derived cross-page artefacts whose markers do not match the tier 1 regex shape and therefore require a plugin-specific fusion step.

The consolidation session of 17 May 2026 (commit `c01661e`) reinforces this rinvio with a fresh empirical observation. After the body+note glued splitter described in pattern (u) lifts Vol. III's NOTE count from 4 to 466, the corpus still falls short of the 744-marker expectation derived from the CROSS_REFERENCE total of 1531: the residual ~278-node delta is attributable mostly to cross-page note continuations whose second-page fragment is itself a glued block and therefore has no leading `(N)` marker that the tier 1 resolver can latch onto. Promoting `merge_cross_page_notes` from placeholder to a real step — one that walks pairs of consecutive pages, detects the orphan continuation of a note interrupted by a page break, and fuses it back into its head — would lift the count further and close most of the delta. The promotion is documented here as the next strong candidate but remains intentionally rinviata to a dedicated session, because the heuristic must be designed against the full four-volume series rather than against the single Vol. III fixture, and that calibration work is too substantial to retrofit at the tail of a consolidation session.

**Propagating new `Node` fields through the pipeline.** When a future schema bump adds a new field to `Node` in `reconstruction/types.py`, three call sites must propagate it explicitly because they rebuild the node from a private `_NodeBuilder`: (1) `_NodeBuilder` + `to_frozen()` in `reconstruction/tier1.py`, (2) `_NodeBuilder` + `to_frozen()` + `_thaw_node()` in `apparatus/resolver.py`, (3) the `_convert_node` mapping in `emission/converter.py`. Post-processing steps that use `dataclasses.replace(node, ...)` (e.g. `postprocessing/steps/dehyphenate.py`) preserve unchanged fields automatically and need no update. Both the `summary_items` field added in 0.3.0 and the `toc_items` field added in 0.4.0 needed all three updates; the integration test on the real corpus fixture is what catches missing propagation, where the schema drift test cannot reach. A future field bump session should add the field-propagation check to its definition of done.

## Schema discipline

The JSON schema at `shared/schema.json`, version 0.4.0, is the binding contract between Layer 1 (the Python pipeline) and Layer 2 (the React Native app) as of today. Its authoritative source are the Pydantic v2 models in `pipeline/src/scabopdf_pipeline/schema/contract.py`; the file `shared/schema.json` is regenerated from them via `pipeline/scripts/generate_schema.py` and must never be edited by hand. The narrative reference, with field-by-field semantics and a long section on the rules that govern the whole arrangement, lives in `docs/SCHEMA_v0.4.0.md` (with `docs/SCHEMA_v0.3.0.md`, `docs/SCHEMA_v0.2.0.md` and `docs/SCHEMA_v0.1.0.md` kept as historic baselines); the per-version delta lives in `docs/SCHEMA_CHANGELOG.md`; the SemVer policy for the schema lives in `docs/json-schema-versioning.md`.

Any session that modifies production code capable of changing the shape of Layer 1's output is responsible for verifying conformance with the contract. The sensitive surfaces are concrete and finite: `pipeline/src/scabopdf_pipeline/extraction/types.py`, `pipeline/src/scabopdf_pipeline/classification/types.py`, `pipeline/src/scabopdf_pipeline/reconstruction/types.py`, `pipeline/src/scabopdf_pipeline/apparatus/types.py`, `pipeline/src/scabopdf_pipeline/postprocessing/types.py`, `pipeline/src/scabopdf_pipeline/profiling/profile.py`, `pipeline/src/scabopdf_pipeline/schema/categories.py`, and the corpus plugins under `pipeline/src/scabopdf_pipeline/profiles/` whenever they introduce new categories, new node fields or new warning vocabulary that the contract must mirror. The emission converter at `pipeline/src/scabopdf_pipeline/emission/converter.py` is the symmetrical surface that translates the Python representation into the contract: it sees both sides of the boundary and must stay in lockstep with every change to the upstream types. A change to any of these — adding a field, removing one, renaming one, changing a type, altering cardinality, or extending the set of values of a closed enum — requires an explicit conformance check against `contract.py` performed during the same session that introduces the change. The alignment is never deferred to a later session: deferring it is precisely what produces silent drift between code and contract.

The operational pattern is precise and goes in seven steps. First, modify the production code. Second, update `contract.py` to reflect the new shape of the output, including the `Literal` of `schema_version` if the bump is minor or major. Third, update `converter.py` to populate any new fields or drop fields that have been removed — the converter is the bridge that the drift test cannot see, and the integration test on Patriarca/Mosconi/Tesauro is what catches a converter that has fallen behind. Fourth, regenerate `shared/schema.json` by running `python pipeline/scripts/generate_schema.py` from the repository root. Fifth, update `docs/SCHEMA_v0.4.0.md` (or the current-version successor) to describe the change narratively. Sixth, update `docs/SCHEMA_CHANGELOG.md` with the motivation, the list of fields added/modified/removed, and a `BREAKING:` line if it applies. Seventh, run the full test suite and verify that every test passes — including the drift test in `pipeline/tests/unit/schema/test_generate_schema.py` and the end-to-end integration tests. The drift test regenerates the schema in memory from the current `contract.py` and compares it byte-for-byte with the committed `shared/schema.json`; if the two differ the test fails with a message that reminds you to rerun the generator. Treat the drift test as the schema-side gate.

The contract has a third line of defence active in production since the post-audit consolidation of 13 May 2026 (commit `615173e`, Decision H of the audit). The emission boundary in `pipeline/src/scabopdf_pipeline/emission/emitter.py` and the CLI's `_run_pipeline` in `pipeline/src/scabopdf_pipeline/emission/cli.py` now run **two independent validators in sequence** under the same `validate=True` flag of `emit_to_file`: first `validate_document` parses the dumped dict against the Pydantic models (which is what `contract.py` describes), then `validate_against_schema` validates the same dict against the committed `shared/schema.json` read from disk with `jsonschema`. The committed schema is loaded once per process through an `@lru_cache(maxsize=1)`-decorated helper, so the disk read and JSON parsing cost is amortised. If the two validators ever disagree — because Pydantic's `model_json_schema()` emits something subtly different from what its own `model_validate` enforces, or because the committed schema on disk has drifted from `contract.py` between regenerations — the second check catches it at runtime as a `jsonschema.ValidationError` and the emitter wraps it into an `EmissionError` with `__cause__` preserved. The flag is meant to stay on in production: turning it off (`validate=False` on `emit_to_file`, `--no-validate` on the CLI) is reserved for emergencies and benchmarks.

This discipline is non-negotiable. It must hold under the pressure of long sessions, unexpected discoveries mid-implementation, time constraints, or any reason that might make skipping a step feel reasonable. The schema is useful only insofar as Layer 1's emitted output and `contract.py` describe the same thing: the moment they diverge, the contract becomes a quiet lie, the test suite stops protecting the boundary it was built to protect, and the next session that runs the pipeline end-to-end inherits a bug whose origin will be buried under intervening commits. Section 6 of `docs/SCHEMA_v0.4.0.md` is the extended reference for this discipline and must be consulted before any structural change to the contract or to the production surfaces listed above.
